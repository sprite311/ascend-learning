# P5C18 — 模型导出与格式转换：ONNX → OM 的旅程

## 引子

要在昇腾上高性能推理，通常需要把模型从 PyTorch 格式转换为昇腾专用的 .om 格式。这个转换过程中经常遇到各种"不兼容"问题，理解背后的逻辑才能高效解决。

---

## 1. 为什么需要格式转换

PyTorch 模型是 Python 代码 + 权重文件的组合，它依赖 Python 运行时来解释和执行。这对于生产推理有两个问题：

1. **Python 太慢**：GIL 锁、动态解释、无法深度优化
2. **运行时依赖**：需要完整的 PyTorch + Python 环境

解决方案：把模型导出为**与框架无关的中间格式**（如 ONNX），再编译为**硬件专用的优化格式**（如 .om）。

---

## 2. 导出链路

```
PyTorch 模型 (.pt/.bin)
      │
      ▼  torch.onnx.export()
ONNX 模型 (.onnx)         ← 通用中间格式
      │
      ▼  ATC 编译器
昇腾优化模型 (.om)          ← 硬件专用格式
```

### 2.1 ONNX：通用中间格式

ONNX（Open Neural Network Exchange）是一个开放的模型交换格式。它把模型表示为一张**计算图**（节点是算子，边是张量），与具体框架无关。

ONNX 是"中立翻译"——从 PyTorch 翻译成通用语言，再从通用语言翻译成昇腾语言。

### 2.2 .om：昇腾的优化产物

.om（Offline Model）是经过 ATC 编译器全面优化后的模型文件。它包含：
- 优化后的计算图（算子融合、常量折叠等已完成）
- 针对昇腾硬件的数据布局（NC1HWC0 等）
- 内存分配计划
- 可直接在 NPU 上执行的指令序列

---

## 3. 导出常见问题

### 3.1 动态 Shape 处理

PyTorch 模型通常支持动态输入（不同长度的文本、不同尺寸的图片）。但 ATC 编译默认需要固定形状。

解决方案：
- **固定形状编译**：指定具体的 batch_size 和 seq_len，性能最优但灵活性差
- **动态 Shape**：指定形状范围（如 `batch:[1,32], seq:[1,2048]`），ATC 会生成通用代码

### 3.2 算子不支持

ONNX 导出时可能遇到 PyTorch 操作无法转换为 ONNX 标准算子的情况。常见的有：
- 自定义 Python 操作
- torch.compile 生成的特殊操作
- 某些容器操作（如动态列表）

ATC 编译时可能遇到 ONNX 算子在 CANN 中不支持的情况。

### 3.3 两种推理路径

需要注意，大模型推理（LLM）通常**不走 ONNX → .om 这条路**。因为 LLM 的自回归生成需要动态管理 KV Cache，这超出了静态计算图的能力。

LLM 推理通常走：
```
PyTorch 模型 → torch_npu 直接运行
            或 → 推理引擎（MindIE/vLLM）动态调度
```

ONNX → .om 路径更适合**传统模型**（CNN、BERT 等固定输入输出的模型）。

---

## 4. LLM 推理的模型加载方式

对于大语言模型，典型的加载方式是：

```
HuggingFace 格式权重 (.safetensors/.bin)
      │
      ▼  推理引擎加载
MindIE 或 vLLM 内部管理
      │
      ▼  torch_npu + CANN
NPU 上执行
```

推理引擎直接读取 HuggingFace 格式的权重，在内部完成模型构建和优化。不需要经过 ONNX 转换。

## 本章小结

| 要点 | 内容 |
|------|------|
| 格式转换目的 | 脱离 Python 运行时，获得硬件级优化 |
| 两条路径 | 传统模型走 ONNX → .om；LLM 走推理引擎直接加载 |
| ONNX 作用 | 框架无关的中间表示 |
| .om 作用 | 包含所有优化的硬件专用格式 |
| 常见问题 | 动态 Shape、算子不支持 |
