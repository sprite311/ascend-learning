# P4C15 — 在昇腾上训练模型：从单卡到集群

## 引子：训练 vs 推理——两个截然不同的世界

前面我们一直在讲模型的推理部署。但很多场景下，你还需要在昇腾上**训练**模型——微调一个预训练模型来适配你的业务，或者从头训练一个小模型。

训练和推理虽然跑的是同一个模型架构，但在计算特性、内存需求和优化策略上有根本区别。

---

## 1. 训练与推理的核心差异

| 维度 | 训练 | 推理 |
|------|------|------|
| 计算方向 | 前向 + 反向传播 | 仅前向 |
| 内存需求 | 极高（需保存中间激活值、梯度、优化器状态） | 较低（只保存权重和 KV Cache） |
| 计算密度 | 高（大 Batch、完整序列） | 低（自回归逐 Token 生成） |
| 主要瓶颈 | Compute-Bound（算力限制） | Memory-Bound（带宽限制） |
| 精度要求 | 需要高精度（梯度更新敏感） | 可量化（INT8/INT4 可接受） |

**一句话理解**：训练是"算力不够用"，推理是"带宽不够用"。

---

## 2. 单卡训练

### 2.1 基本流程

在昇腾上做单卡训练的流程和 GPU 上完全相同（以 PyTorch + torch_npu 为例）：

```python
import torch
import torch_npu

model = MyModel().npu()            # 模型放到 NPU
optimizer = torch.optim.Adam(model.parameters(), lr=1e-4)

for batch in dataloader:
    inputs = batch['input'].npu()   # 数据放到 NPU
    labels = batch['label'].npu()
    
    outputs = model(inputs)          # 前向传播
    loss = criterion(outputs, labels)
    
    optimizer.zero_grad()
    loss.backward()                  # 反向传播
    optimizer.step()                 # 更新参数
```

### 2.2 混合精度训练

大模型训练几乎都使用混合精度（FP16/BF16 + FP32）来节省显存和加速：

- **前向和反向传播**：使用 FP16/BF16（速度快、省显存）
- **权重更新**：使用 FP32（保证精度）

昇腾支持 BF16（Brain Float 16），它比 FP16 数值范围更大，在训练中更不容易溢出。

### 2.3 单卡显存估算

训练一个模型需要的显存远超推理。一个粗略的估算公式：

```
训练显存 ≈ 模型参数 × (2 + 2 + 12) = 参数量 × 16 字节

其中：
- 2 字节：FP16 模型参数
- 2 字节：FP16 梯度
- 12 字节：FP32 优化器状态（Adam 需要 momentum + variance + FP32 权重副本）
```

| 模型大小 | 推理显存（FP16） | 训练显存（Adam + AMP） |
|---------|---------------|---------------------|
| 1B | 2 GB | 16 GB |
| 7B | 14 GB | ~112 GB |
| 13B | 26 GB | ~208 GB |
| 70B | 140 GB | ~1120 GB |

一张 Ascend 910B（64 GB HBM）可以训练 ~4B 参数的模型（全量训练）。大于这个规模就需要分布式训练或参数高效微调（LoRA 等）。

## 3. 分布式训练

### 3.1 为什么需要分布式

上面的估算告诉我们：7B 模型的全量训练需要约 112 GB 显存，单张 910B（64 GB）放不下。解决方案就是把工作量分摊到多张卡上。

### 3.2 三种并行策略

**数据并行（Data Parallelism, DP）**

每张卡都有完整的模型副本，但处理不同的数据批次。每个小批次计算完梯度后，用 AllReduce 通信把所有卡的梯度同步。

- 适用：模型放得下一张卡，想提高训练速度
- 显存节省：不节省（每张卡都有完整模型）
- 通信量：中等（梯度同步）

**张量并行（Tensor Parallelism, TP）**

把单个算子（如一个矩阵乘法）的权重矩阵切分到多张卡上。每张卡只做矩阵的一部分计算。

- 适用：模型太大，单卡放不下
- 显存节省：线性降低
- 通信量：较大（每一层都需要通信）

**流水线并行（Pipeline Parallelism, PP）**

把模型按层切分——前 10 层在卡 0，中间 10 层在卡 1，后 10 层在卡 2。数据像流水线一样依次经过。

- 适用：模型很深（层数多）
- 显存节省：线性降低
- 通信量：较小（只在切分点通信）

### 3.3 混合并行

实际大模型训练通常**组合使用**多种策略：

```
8 台机器 × 8 卡/台 = 64 张卡

PP = 4 级（模型按层分成 4 段）
TP = 4 路（每段内 4 张卡做张量并行）
DP = 4 路（4 组相同配置处理不同数据）

4 × 4 × 4 = 64 张卡
```

### 3.4 在昇腾上配置分布式训练

使用 PyTorch + torch_npu 的分布式训练：

```python
import torch.distributed as dist
import torch_npu

# 初始化通信（使用 HCCL 后端）
dist.init_process_group(backend='hccl')

# 设置设备
local_rank = int(os.environ['LOCAL_RANK'])
torch.npu.set_device(local_rank)

# 模型并行化
model = MyModel().npu()
model = torch.nn.parallel.DistributedDataParallel(model)
```

底层通信使用 HCCL（对标 NCCL），机内走 HCCS 高速互联，机间走 RoCE 网络。

## 4. 参数高效微调（PEFT）

全量训练 7B 以上的模型对显存要求极高。更实际的做法是使用**参数高效微调**技术，只更新少量参数：

### 4.1 LoRA

LoRA（Low-Rank Adaptation）是目前最流行的高效微调方法。核心思想：

- 冻结原始模型权重（不更新）
- 在关键层旁边加入两个很小的矩阵（低秩分解）
- 只训练这两个小矩阵

效果：7B 模型用 LoRA 微调，显存需求降到 ~16 GB，**单张 910B 轻松搞定**。

### 4.2 在昇腾上使用 LoRA

通过 HuggingFace PEFT 库 + torch_npu：

```python
from peft import LoraConfig, get_peft_model

config = LoraConfig(r=16, lora_alpha=32, target_modules=["q_proj", "v_proj"])
model = get_peft_model(base_model, config).npu()
# 正常训练流程...
```
