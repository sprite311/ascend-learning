# P2C04 — 达芬奇架构总览：一颗 AI 芯片的内部长什么样

## 引子：打开芯片的"外壳"

上一部分我们知道了昇腾的核心芯片使用"达芬奇架构"。但"达芬奇架构"到底是什么意思？芯片里面是怎么工作的？

如果你只把芯片当成一个黑盒（输入数据进去、结果出来），在模型跑得正常的时候没什么问题。但当你遇到以下情况时，就必须打开这个黑盒：

- 模型跑得比预期慢 3 倍——到底是哪个环节在拖后腿？
- 报了 OOM（内存溢出）——到底是哪一级存储满了？
- 性能分析报告上写着"Cube utilization 12%"——这意味着什么？

本章带你走进芯片内部，建立一个清晰的硬件心智模型。

---

## 1. AI Core：达芬奇架构的基本单元

一颗昇腾芯片（比如 Ascend 910B）内部不是一个单一的计算引擎，而是由**多个 AI Core** 组成的阵列。

你可以把一颗芯片想象成一家工厂：
- 整颗芯片 = 工厂
- 每个 AI Core = 一个独立的**车间**
- 一颗 Ascend 910B 大约有 **30+ 个车间**同时运转

每个车间（AI Core）内部有以下"工种"：

```
┌──────────────────────────────────────────┐
│                AI Core                    │
│                                          │
│  ┌──────────┐  ┌──────────┐  ┌────────┐ │
│  │  Cube    │  │  Vector  │  │ Scalar │ │
│  │ 矩阵运算 │  │ 向量运算  │  │ 流程控制│ │
│  └──────────┘  └──────────┘  └────────┘ │
│                                          │
│  ┌──────────────────────────────────────┐ │
│  │          片上缓存 (SRAM)              │ │
│  │     L1 Buffer / L0 Buffer            │ │
│  └──────────────────────────────────────┘ │
│                                          │
│  ┌──────────────────────────────────────┐ │
│  │        MTE (数据搬运引擎)             │ │
│  │   负责在 HBM ↔ SRAM 之间搬运数据     │ │
│  └──────────────────────────────────────┘ │
└──────────────────────────────────────────┘
```

让我们逐一了解每个"工种"。

## 2. Cube 单元：矩阵运算的心脏

### 2.1 它做什么

Cube（立方体）是达芬奇架构最引以为傲的部件。它的唯一职责就是：**做矩阵乘法**。

具体来说，Cube 在**一个时钟周期**内可以完成一个 **16×16 矩阵和一个 16×16 矩阵**的乘法累加运算。

一次运算 = 16 × 16 × 16 = **4,096 次乘加操作**。

### 2.2 为什么叫"Cube"

普通的乘法是一维的（标量 × 标量 = 标量）。矩阵乘法是二维的（行 × 列 = 元素）。而 Cube 的计算阵列在物理上是**三维排布**的——输入 A 从一个方向流入，输入 B 从另一个方向流入，结果 C 从第三个方向流出。像一个立方体，所以叫 Cube。

### 2.3 为什么是 16×16？

这不是一个任意选择，而是多重约束下的平衡：

| 因素 | 越大越好？ | 越小越好？ |
|------|----------|----------|
| 计算吞吐 | ✓ 阵列越大，一次算的越多 | |
| 数据利用率 | | ✓ 阵列越大，越难填满（小矩阵"浪费"） |
| 芯片面积 | | ✓ 越大占用越多硅片面积 |
| 功耗 | | ✓ 越大功耗越高 |

16×16 是当前工艺和 AI 计算负载下的一个**甜蜜点**。对于大模型的矩阵运算（维度通常是 4096、8192 这样的大数字），16×16 的分块可以被高效地反复调用来完成整个大矩阵的计算。

### 2.4 Cube 的"挑食"问题

Cube 虽然计算能力极强，但它有一个重要特性：**对数据格式非常挑剔**。

它不接受人类常用的矩阵格式（如 NCHW），而要求数据必须按照特定的**分形格式**（如 NC1HWC0）排列。

为什么？因为在硬件层面，Cube 的数据通路是按特定的排列方式物理连线的。数据如果不按这个格式来，就像把圆形零件塞进方形接口——要么塞不进去，要么需要额外的加工（TransData 操作），白白浪费时间。

这个问题的解决方案不在硬件层（让用户手工转格式不现实），而是在软件层——图编译器会自动插入格式转换操作。这是 P3 会讲的内容。

## 3. Vector 单元：精细活儿的能手

### 3.1 它做什么

AI 模型不是只有矩阵乘法。在矩阵乘法之间，还夹杂着大量的"精细操作"：

- **Softmax**：计算注意力权重时要做指数运算和归一化
- **LayerNorm**：每一层之后要做均值和方差的计算
- **GELU/SiLU**：激活函数，是逐元素的非线性运算
- **Add**：残差连接中的逐元素加法

这些操作有一个共同特点：**不是矩阵乘法，而是逐元素或逐向量的操作**。让 Cube 来干这活，就像用吊车搬鸡蛋——太浪费。

Vector 单元就是专门做这些精细活的。它一次可以对一整排数字（一个向量）做运算，效率远高于逐个处理。

### 3.2 Cube 和 Vector 的协作

在一个 Transformer 层的计算过程中，Cube 和 Vector 的协作就像接力赛：

```
输入 → [Cube: QKV矩阵乘] → [Vector: Softmax] → [Cube: Attention输出矩阵乘]
     → [Vector: Add残差] → [Vector: LayerNorm] → [Cube: FFN第一层矩阵乘]
     → [Vector: GELU激活] → [Cube: FFN第二层矩阵乘] → [Vector: Add残差]
     → [Vector: LayerNorm] → 输出
```

可以看到，Cube 和 Vector 是**交替执行**的。这种"接力"能否流畅运行，直接决定了整体性能。如果 Vector 处理太慢，Cube 就得等着，反之亦然。

## 4. Scalar 单元：调度指挥官

### 4.1 它做什么

Scalar 单元负责的不是"计算"，而是"指挥"：

- 计算内存地址（数据在缓存的什么位置？）
- 控制程序流程（循环、条件判断）
- 管理同步（Cube 算完了，通知 Vector 可以开始）

### 4.2 为什么需要它

如果没有 Scalar，Cube 和 Vector 就像两个埋头干活但不知道接下来该干什么的工人。Scalar 就是车间主任，告诉它们："数据在内存第 X 个位置，算完放到第 Y 个位置，算了 N 次之后停下来。"

## 5. 三者的协同：流水线设计

达芬奇架构的真正精妙之处，不在于单个单元有多强，而在于三者如何协同工作。

### 5.1 流水线思想

想象一家洗车店：
- **洗车**需要 3 步：冲水（5 分钟）→ 打泡沫搓洗（5 分钟）→ 吹干（5 分钟）
- 如果一辆一辆来，每辆车 15 分钟，3 辆车需要 45 分钟
- 但如果用流水线：第 1 辆在搓洗时，第 2 辆开始冲水；第 1 辆吹干时，第 2 辆搓洗，第 3 辆冲水。3 辆车只需要 25 分钟

AI Core 内部的 Cube、Vector、Scalar 也是按流水线方式运作的：

```
时间 →
Scalar: [地址计算1] [地址计算2] [地址计算3] ...
MTE:    [搬运数据1] [搬运数据2] [搬运数据3] ...
Cube:              [矩阵乘1]  [矩阵乘2]  [矩阵乘3]  ...
Vector:                       [后处理1]  [后处理2]  [后处理3] ...
```

当 Cube 在计算第一批数据的矩阵乘法时，MTE（数据搬运引擎）已经在搬运第二批数据了，Scalar 已经在计算第三批数据的地址了。

### 5.2 双缓冲（Double Buffering）

为了让流水线不中断，片上缓存被分成了**两块**：
- Cube 从**缓冲区 A** 读取数据计算时
- MTE 同时往**缓冲区 B** 搬运下一批数据
- 计算完成后，AB 角色互换

这就像餐厅的"备菜"机制：厨师在炒当前这盘菜时，备菜师傅已经把下一盘菜的原料切好、摆好，放在旁边等着了。

## 6. 片上资源总览

一个 AI Core 内部的完整资源清单：

| 资源 | 大小（典型值） | 作用 |
|------|-------------|------|
| Cube 单元 | 1 组 16×16×16 MAC 阵列 | 矩阵乘法 |
| Vector 单元 | 多条 SIMD 向量流水线 | 向量/标量运算 |
| Scalar 单元 | 1 个标量处理器 | 流程控制 |
| L0 Buffer | 几百 KB | Cube 的输入/输出缓冲 |
| L1 Buffer | 约 1 MB | AI Core 的本地缓存 |
| L2 Cache | 数 MB（多 Core 共享） | 芯片级共享缓存 |
| MTE | 数据搬运引擎 | HBM ↔ L1 ↔ L0 搬运 |

而整颗芯片的资源：

| 资源 | 典型值（Ascend 910B） |
|------|---------------------|
| AI Core 数量 | 30+ |
| HBM 容量 | 64 GB |
| HBM 带宽 | ~1.6 TB/s |
| FP16 峰值算力 | ~320 TFLOP/s |
| 互联接口 | HCCS（芯片间高速互联）|
| TDP | ~310W |

## 7. 建立硬件心智模型

学完本章后，当你看到性能分析报告时，你应该能在脑中形成这样的画面：

1. **"Cube utilization 12%"** → 矩阵计算单元大部分时间在空转，可能是数据搬运太慢（Memory-Bound），或者 Vector 操作太多拖后腿

2. **"大量 TransData 算子"** → 数据格式在不同层之间来回转换，Cube 挑食的问题没有被编译器完美解决

3. **"L1 miss rate 高"** → 数据在片上缓存中放不下或被反复驱逐，需要检查 Tiling（分块）策略

4. **"HBM bandwidth 打满"** → 芯片在疯狂搬数据但计算不够多，典型的 Memory-Bound，考虑量化或算子融合

## 本章小结

| 要点 | 内容 |
|------|------|
| 芯片结构 | 由多个 AI Core 组成（30+ 个） |
| AI Core 三单元 | Cube（矩阵乘）+ Vector（向量运算）+ Scalar（流程控制） |
| 核心设计 | 16×16×16 MAC 阵列，一个周期 4096 次乘加 |
| 协同方式 | 流水线 + 双缓冲，计算和搬运并行 |
| 关键约束 | Cube 对数据格式挑剔（NC1HWC0 分形格式） |
| 存储层次 | L0 → L1 → L2 → HBM，越近越快越小 |
