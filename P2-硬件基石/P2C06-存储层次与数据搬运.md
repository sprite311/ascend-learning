# P2C06 — 存储层次与数据搬运：为什么优化常是"解决搬砖"

## 引子：最反直觉的优化经验

很多初次在昇腾（或 GPU）上做性能优化的工程师，都经历过这样一件困惑的事：

费了很大力气把某个算子的计算逻辑优化了 30%，满怀信心地跑测试——结果总时间几乎没变。

为什么？因为**那个算子的时间根本不是被"算"占满的，而是被"搬数据"占满的**。计算提速 30%，但搬运时间不变，而搬运占了 90% 的时间，总时间自然纹丝不动。

这就是 AI 计算中最核心也最反直觉的一个认知：**大多数时候，性能瓶颈不在计算，而在数据搬运。** 理解这一点，需要先理解芯片内部的存储层次。

---

## 1. 存储层次：为什么要分这么多级？

### 1.1 一个生活类比

想象你在图书馆写论文：

| 存储位置 | 类比 | 容量 | 速度 |
|---------|------|------|------|
| **你的脑子**（寄存器） | 正在思考的内容 | 极小（几个想法） | 瞬间 |
| **桌上的书**（L0/L1 缓存） | 翻开摊在面前的 | 小（3-5 本） | 伸手就拿 |
| **旁边书架**（L2 缓存） | 同一排书架上的 | 中（几十本） | 走几步 |
| **图书馆书库**（HBM） | 整个图书馆的藏书 | 大（几十万本） | 要走到地下室 |
| **其他图书馆**（另一张卡/主机内存） | 城市另一头的图书馆 | 巨大 | 要坐车去 |

在写论文的过程中，你最频繁使用的是桌上的书（缓存），偶尔去书架取一本新的，很少去书库，几乎不去其他图书馆。

芯片的设计也遵循同样的逻辑。

### 1.2 昇腾的存储层次

```
速度（快→慢）
    ↑
    │  ┌─────────────┐
    │  │  寄存器/L0   │  ← 几百 KB，纳秒级，紧贴 Cube/Vector
    │  ├─────────────┤
    │  │  L1 Buffer   │  ← ~1 MB/Core，纳秒级，AI Core 本地
    │  ├─────────────┤
    │  │  L2 Cache    │  ← 数 MB，十纳秒级，多 Core 共享
    │  ├─────────────┤
    │  │  HBM         │  ← 64 GB，百纳秒级，高带宽主存
    │  ├─────────────┤
    │  │  Host 内存    │  ← 数百 GB，微秒级，CPU 侧 DDR
    │  └─────────────┘
    │
    └──→ 容量（小→大）
```

**核心规律：越靠近计算单元，速度越快，但容量越小。**

这不是设计缺陷，而是物理定律决定的。快速存储（SRAM）的每一位都需要 6 个晶体管，而大容量存储（DRAM/HBM）每一位只需要 1 个晶体管 + 1 个电容。要让存储又大又快，成本和功耗会爆炸式增长。

## 2. HBM：高带宽但不够快

### 2.1 什么是 HBM

HBM（High Bandwidth Memory，高带宽内存）是 AI 芯片的"主仓库"，存放模型的所有权重参数和中间计算结果。

以 Ascend 910B 为例：
- 容量：64 GB
- 带宽：约 1.6 TB/s

"1.6 TB/s"听起来已经很快了——每秒能传输 1.6 万亿字节的数据。但让我们看看实际需求。

### 2.2 算术强度：判断瓶颈的关键指标

**算术强度（Arithmetic Intensity）** = 计算量（FLOP）/ 数据搬运量（Bytes）

它衡量的是：每搬运 1 字节数据，芯片能做多少次计算。

昇腾 910B 的"平衡点"大约是：
- 峰值算力 320 TFLOP/s ÷ HBM 带宽 1.6 TB/s = **200 FLOP/Byte**

这意味着：**如果一个操作的算术强度低于 200，芯片就是带宽瓶颈（Memory-Bound）；高于 200，才是算力瓶颈（Compute-Bound）。**

### 2.3 大模型推理的残酷现实

大模型推理分两个阶段：

| 阶段 | 操作 | 算术强度 | 瓶颈类型 |
|------|------|---------|---------|
| **Prefill**（填充） | 一次处理整个输入序列 | 高（~数千） | Compute-Bound |
| **Decode**（生成） | 逐 Token 生成 | 极低（~1-2） | Memory-Bound |

Decode 阶段的算术强度只有 1-2，远低于 200 的平衡点。为什么？

因为生成每个 Token 时，模型需要把**全部几十 GB 的权重参数**从 HBM 读取到计算单元，但每次只做一个 Token 的计算（一个向量乘以整个权重矩阵，即 GEMV）。

搬运了 14 GB 的数据（7B 模型、FP16），只做了 14 GFLOP 的计算。算术强度 = 14G / 14G = 1。

这就是为什么大模型推理的生成速度不取决于芯片有多少算力，而取决于**内存带宽有多高**。

## 3. 数据搬运引擎：MTE

### 3.1 MTE 是什么

MTE（Memory Transfer Engine，数据搬运引擎）是 AI Core 内部专门负责数据搬运的硬件单元。它的工作方式类似于"快递员"：

- 从 HBM 取数据放到 L1 缓存
- 从 L1 搬到 L0 给 Cube/Vector 使用
- 把计算结果从 L0 搬回 L1，再写回 HBM

### 3.2 关键特性：异步搬运

MTE 和计算单元是**独立工作**的。也就是说，MTE 在搬运第二批数据的同时，Cube 可以计算第一批数据。这种并行是性能的关键。

```
时间轴 →

MTE:   [搬运批次1] [搬运批次2] [搬运批次3]
Cube:              [计算批次1] [计算批次2]
```

如果搬运时间 ≤ 计算时间，MTE 的搬运就被完全"隐藏"了——从外部看，就好像数据瞬间就到了。

但如果搬运时间 > 计算时间（Memory-Bound 场景），Cube 就会出现空闲等待。

---

## 4. 三大应对策略

理解了"搬砖比算砖慢"的根本原因后，应对策略就自然而然了：

### 4.1 策略一：减少搬运体积——量化

把数据从 FP16（每个数占 2 字节）压缩到 INT8（1 字节）甚至 INT4（0.5 字节）。

**效果**：一个 7B 模型从 14 GB 缩小到 7 GB（INT8）或 3.5 GB（INT4）。搬运时间直接减半或减到四分之一。

**代价**：精度损失。但在推理场景下，通过适当的量化校准技术，精度损失通常可以控制在可接受范围内。

### 4.2 策略二：搬一次算多份——Batching

既然搬一次权重那么贵，不如把多个用户的请求拼在一起。把权重搬进来后，同时算 8 个人的结果。

**效果**：搬运成本被 8 个用户平摊，单用户等效带宽提升 8 倍。

从算术强度看：单请求是 GEMV（向量×矩阵，算术强度~1），8 个请求拼在一起变成 GEMM（矩阵×矩阵，算术强度~8），离平衡点更近了。

**代价**：单个请求的延迟可能增加（要等凑够一批），但吞吐量大幅提升。

### 4.3 策略三：减少搬运次数——算子融合 & KV Cache

**算子融合**：假设计算过程是 MatMul → Add → LayerNorm。如果分开做，MatMul 的结果要写回 HBM，Add 再从 HBM 读出来……三次搬运。如果融合成一个算子，结果留在缓存里一路做到底，只需要一次搬运。

**KV Cache**：在 Transformer 的自回归生成中，每生成一个新 Token，之前所有 Token 的 Key 和 Value 需要重新参与计算。如果每次都重新算，计算量和搬运量都很大。KV Cache 把历史的 Key/Value 缓存起来，每次只算新 Token 的部分。

## 5. 一个完整的例子

让我们用一个具体例子把所有概念串起来：

**场景**：在 Ascend 910B 上，用一个 7B 参数的 FP16 模型，做单请求推理生成。

| 参数 | 数值 |
|------|------|
| 模型大小 | 14 GB（FP16） |
| HBM 带宽 | 1.6 TB/s |
| FP16 峰值算力 | 320 TFLOP/s |
| 生成一个 Token 的计算量 | ~14 GFLOP |
| 生成一个 Token 需搬运的数据 | ~14 GB |

**搬运时间**：14 GB ÷ 1.6 TB/s = **8.75 ms**
**计算时间**：14 GFLOP ÷ 320 TFLOP/s = **0.04 ms**

搬运时间是计算时间的 **200 多倍**！芯片 99.5% 的时间在搬数据，只有 0.5% 的时间在计算。

**每秒能生成多少 Token？** 约 1000 / 8.75 ≈ **114 Token/s**（理论上限）。

如果用 INT8 量化（模型缩到 7 GB），搬运时间减半，理论上限翻倍到 **228 Token/s**。

如果同时 Batch 8 个请求（仍用 FP16），搬运开销被 8 个请求分摊，单请求等效速度也翻倍。

## 本章小结

| 要点 | 内容 |
|------|------|
| 存储层次 | L0 → L1 → L2 → HBM → Host，越快越小 |
| 算术强度 | 计算量/搬运量，低于芯片平衡点就是 Memory-Bound |
| 推理的残酷现实 | Decode 阶段算术强度极低（~1-2），严重的 Memory-Bound |
| MTE 搬运引擎 | 和计算异步并行，但受 HBM 带宽上限制约 |
| 三大应对 | 量化（减体积）、Batching（摊成本）、算子融合（减次数） |
