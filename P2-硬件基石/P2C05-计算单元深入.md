# P2C05 — Cube 与 Vector：计算单元的分工逻辑

## 引子：为什么不能只用一种计算单元？

你可能会问：既然 Cube 这么强大（一个周期 4096 次乘加），为什么不把整个 AI Core 都做成 Cube？或者反过来，Vector 更通用，为什么不全用 Vector？

答案在于：**AI 计算中不同操作的计算模式有本质区别，一种单元无法高效处理所有类型的运算。**

这就像厨房里需要大灶和精细料理台：大灶做大锅炒菜效率高，但你不能用大灶来切刺身。

---

## 1. 两种运算模式的根本差异

### 1.1 矩阵运算 vs 向量运算

让我们从数学层面看看差异：

**矩阵运算**（Cube 负责）：
```
输入 A: [M × K] 矩阵
输入 B: [K × N] 矩阵
输出 C: [M × N] 矩阵
操作：C[i][j] = Σ A[i][k] × B[k][j]  (k=1到K)
```
特点：**每个输出元素都依赖输入的多个元素**，存在大量的数据复用（A 的一行被所有列用到，B 的一列被所有行用到）。

**向量运算**（Vector 负责）：
```
输入 X: [N] 向量
输出 Y: [N] 向量
操作：Y[i] = f(X[i])  每个元素独立变换
```
特点：**每个输出元素只依赖对应位置的输入元素**，没有跨元素的数据复用。

### 1.2 为什么需要不同的硬件

| 特性 | 矩阵运算 | 向量运算 |
|------|---------|---------|
| 数据复用率 | 极高（一份数据被反复使用） | 极低（一份数据只用一次） |
| 计算密度 | 高（计算量/数据量 比大） | 低（计算量/数据量 比小） |
| 适合的硬件 | 大规模并行阵列（Cube） | 灵活的流水线（Vector） |
| 瓶颈 | 计算能力 | 内存带宽 |

如果用 Cube 做向量运算（比如 GELU 激活函数）：
- Cube 期望的输入是两个矩阵，但激活函数只有一个向量输入
- 大量计算单元闲置，利用率可能低于 5%
- 相当于开了一整条生产线只为了拧一个螺丝

如果用 Vector 做矩阵运算：
- Vector 没有矩阵乘法的专用硬件，只能把矩阵乘法拆成一行一行的向量运算
- 失去了数据复用的优势，性能下降 10-100 倍
- 相当于手工一个一个缝扣子，不用缝纫机

## 2. Cube 深入：矩阵乘法的硬件实现

### 2.1 计算过程可视化

以一个简化的例子说明 Cube 如何做矩阵乘法：

假设要计算 C = A × B，其中 A 是 [4×4]，B 是 [4×4]。

在 Cube 内部，A 的数据从左侧流入，B 的数据从上方流入：

```
        B 的列
        ↓ ↓ ↓ ↓
A 的 → [×] [×] [×] [×] → C 的行
行   → [×] [×] [×] [×]
     → [×] [×] [×] [×]
     → [×] [×] [×] [×]
```

每个 `[×]` 是一个乘加单元。在一个时钟周期内，所有单元同时工作，A 的一个元素和 B 的一个元素同时到达交叉点，完成乘法并累加到 C 中。

实际的 Cube 是 16×16×16 的三维阵列，升级版本可支持 32×16×16，原理相同但规模更大。

### 2.2 分块计算（Tiling）

现实中的矩阵通常远大于 16×16（大模型的权重矩阵动辄 4096×4096 甚至 12288×12288）。Cube 怎么处理？

**分块（Tiling）** ——把大矩阵切成 16×16 的小块，一块一块算，最后拼起来。

```
大矩阵 [4096 × 4096] 
= 256 × 256 个 [16×16] 小块
```

Cube 依次（或并行地，多个 AI Core 同时）计算每个小块，再把结果累加起来。

分块策略的好坏直接影响性能：
- **块太小**：每次计算量小，搬运数据的开销占比过大
- **块太大**：超出片上缓存容量，数据放不下
- **最优分块**：让每次搬运的数据在缓存里被充分复用，搬一次用多次

### 2.3 支持的数据类型

Cube 支持多种精度的运算：

| 精度 | 位宽 | 一次 Cube 计算的乘加次数 | 典型用途 |
|------|------|------------------------|---------|
| FP16 | 16 bit | 4,096 | 主流推理/训练 |
| BF16 | 16 bit | 4,096 | 训练（数值范围更大） |
| INT8 | 8 bit | 16,384 | 量化推理（精度换速度）|
| FP32 | 32 bit | 1,024 | 高精度场景 |

注意：**INT8 的吞吐量是 FP16 的 4 倍**。这就是量化推理能加速的根本原因——并不只是模型变小了，而是芯片在同一个周期内能做更多次运算。

## 3. Vector 深入：灵活性的价值

### 3.1 Vector 处理的典型操作

| 操作 | 数学表达 | 出现在模型的哪里 |
|------|---------|----------------|
| Softmax | exp(x_i) / Σexp(x_j) | Attention 权重计算 |
| LayerNorm | (x - μ) / σ | 每一层之后的归一化 |
| GELU | x · Φ(x) | FFN 中的激活函数 |
| Add | a + b | 残差连接 |
| Multiply | a × b | 逐元素缩放 |
| Cast | float16 → float32 | 精度转换 |

### 3.2 Vector 的硬件结构

Vector 单元通常是 **SIMD（Single Instruction Multiple Data）** 架构：一条指令对多个数据同时执行相同操作。

比如 GELU 激活函数：`y[i] = gelu(x[i])`，需要对向量中的每个元素做相同的运算。Vector 单元可以一次性对 256 个（或更多）元素并行计算。

### 3.3 Vector 的"隐形成本"

虽然单个向量操作很快，但**向量操作的总时间在整个模型中的占比可能超出你的预期**。

以一个 Transformer 层为例，假设序列长度 2048，隐藏维度 4096：

| 操作 | 类型 | 计算量（约） |
|------|------|------------|
| QKV 投影 | Cube（矩阵乘） | 100 GFLOP |
| Attention Score | Cube（矩阵乘） | 34 GFLOP |
| Softmax | Vector | 0.03 GFLOP |
| Attention 输出 | Cube（矩阵乘） | 34 GFLOP |
| FFN 第一层 | Cube（矩阵乘） | 67 GFLOP |
| GELU | Vector | 0.03 GFLOP |
| FFN 第二层 | Cube（矩阵乘） | 67 GFLOP |
| LayerNorm ×2 | Vector | 0.01 GFLOP |

从计算量看，Vector 操作微不足道（不到 0.1%）。但在实际执行中，Vector 操作的耗时可能占到 **10-30%**！

为什么？因为 Vector 操作是 **Memory-Bound**（内存带宽瓶颈）——计算量虽小，但每次都要从内存读数据、写结果，搬运时间远大于计算时间。

**这就是为什么算子融合如此重要**：如果能把 MatMul → Add → LayerNorm 三个操作融合成一个，只需要搬运一次数据（而不是三次），性能可能提升 2-3 倍。

## 4. Scalar 单元：被忽视的关键角色

### 4.1 Scalar 在干什么

Scalar 单元不做"数学计算"，而是做"管理工作"：

```
for (int i = 0; i < num_tiles; i++) {
    addr_A = base_A + i * tile_size;     // Scalar: 计算地址
    addr_B = base_B + i * tile_size;     // Scalar: 计算地址
    MTE.move(HBM[addr_A] → L1);         // Scalar: 指挥搬运
    MTE.move(HBM[addr_B] → L1);         // Scalar: 指挥搬运
    Cube.compute(L1_A, L1_B → L0_C);    // Scalar: 触发计算
    MTE.move(L0_C → HBM[addr_C]);       // Scalar: 搬运结果
}
```

Scalar 一直在循环中计算地址、发出指令、协调各个单元的工作节奏。

### 4.2 Scalar 成为瓶颈的场景

如果模型的计算图非常碎片化（大量很小的操作交替执行），Scalar 需要高频切换任务指令，可能成为瓶颈。这种情况在以下场景容易出现：

- 自定义模型中有大量小型操作
- 模型导出时没有做好算子融合
- 动态 shape（每次输入长度不同，Scalar 要反复重新计算参数）

## 5. 三驾马车的性能平衡

### 5.1 理想状态

在理想状态下，Cube、Vector、MTE（搬运）三者完美流水，没有任何一方在等另一方：

```
时间轴 →
MTE:    [搬运1] [搬运2] [搬运3] [搬运4]
Cube:          [计算1] [计算2] [计算3]
Vector:               [后处理1][后处理2]
```

每个时间段都有所有单元在忙碌。这时候芯片的利用率最高。

### 5.2 常见的不平衡

**情况一：Cube 等 MTE（Memory-Bound）**
```
MTE:    [搬运1--------] [搬运2--------]
Cube:   [等...][计算1]  [等...][计算2]
```
搬运太慢，Cube 大部分时间在等数据。在 Profiling 中表现为**Cube utilization 低、HBM bandwidth 接近峰值**。

**情况二：Vector 拖 Cube 后腿**
```
Cube:   [计算1][等待...] [计算2]
Vector: [后处理1------] [后处理2------]
```
Vector 操作太多或太慢，Cube 算完了没法继续。在 Profiling 中表现为 **Vector 时间占比异常高**。

**情况三：Scalar 跑不过来**
```
Scalar: [指令1][指令2][指令3]....(指令发射太慢)
Cube:   [等...][计算1][等...][计算2]
```
任务太碎片化，Scalar 发指令的速度跟不上。在 Profiling 中表现为 **task schedule 时间占比高**。

## 6. 实际影响：为什么懂硬件才能做好优化

理解了 Cube/Vector 的分工后，你就能明白为什么很多优化技巧是这样做的：

| 优化技巧 | 硬件层面的解释 |
|---------|--------------|
| **算子融合** | 减少 Vector 操作之间的内存读写，让数据留在寄存器/缓存中 |
| **量化（INT8/INT4）** | Cube 处理低精度数据吞吐量更高，且搬运体积减半 |
| **大 Batch** | 让 Cube 每次算更大的矩阵，提高计算密度，摊薄搬运成本 |
| **Flash Attention** | 把 Softmax（Vector）和 MatMul（Cube）融合，减少 HBM 读写 |
| **避免碎片操作** | 减少 Scalar 的调度压力 |

## 本章小结

| 要点 | 内容 |
|------|------|
| 为什么两种单元 | 矩阵运算和向量运算的计算模式有本质区别 |
| Cube 特点 | 16×16×16 MAC 阵列，极高吞吐，但挑数据格式 |
| Vector 特点 | SIMD 架构，灵活通用，但常是 Memory-Bound |
| Scalar 作用 | 流程控制和调度，碎片化任务时可能成瓶颈 |
| 性能关键 | 三者的流水线平衡，任何一方拖后腿都会拉低整体性能 |
| 优化方向 | 算子融合、量化、大Batch、减少碎片操作 |
