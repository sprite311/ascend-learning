# P3C10 — 算子与算子库：AI 计算的最小可执行单元

## 引子：为什么"算子"这个概念如此重要

在昇腾的世界里，你经常会听到"算子"这个词：

- "这个模型转换失败了，因为有几个算子不支持"
- "性能瓶颈在 Softmax 算子上"
- "需要开发一个自定义算子"

**算子是连接 AI 模型和芯片硬件的最小桥梁。** 框架层把模型拆解成一个个算子调用，CANN 把每个算子调用翻译成芯片指令。如果某个算子的质量不好（或者根本不存在），就会直接影响模型的精度和性能。

---

## 1. 什么是算子

### 1.1 直观理解

把 AI 模型想象成一个复杂的数学公式，算子就是公式中的**每一步基本运算**。

例如，Transformer 中一个 Self-Attention 的计算可以拆解为：

```
Q = X × W_q        → MatMul 算子
K = X × W_k        → MatMul 算子
V = X × W_v        → MatMul 算子
Score = Q × K^T     → BatchMatMul 算子
Score = Score / √d  → Div 算子
Weights = Softmax(Score)  → Softmax 算子
Output = Weights × V      → BatchMatMul 算子
```

每一行就是一个算子的调用。整个模型由几百到几千个算子调用组成。

### 1.2 算子 vs 函数

你可能觉得"算子不就是函数嘛"。概念上是的，但 AI 领域的算子有一些独特之处：

| 特点 | 普通函数 | AI 算子 |
|------|---------|---------|
| 运行位置 | CPU | NPU/GPU（加速器） |
| 数据类型 | 通用 | 张量（Tensor，多维数组） |
| 优化程度 | 通用实现 | 针对硬件深度优化 |
| 调用方式 | 直接调用 | 通过 Runtime 异步调度 |

### 1.3 算子的分类

按计算特点，算子大致分为以下几类：

| 类别 | 代表算子 | 特点 | 主要使用的硬件 |
|------|---------|------|--------------|
| **计算密集型** | MatMul、Conv2D、BatchMatMul | 大量乘加，计算量大 | Cube |
| **访存密集型** | Softmax、LayerNorm、Add、GELU | 逐元素运算，搬运数据多 | Vector |
| **数据搬运型** | Transpose、Reshape、Concat、Slice | 不做计算，只重排数据 | MTE |
| **通信型** | AllReduce、AllGather | 多卡之间交换数据 | HCCL 通信库 |

## 2. 算子库：CANN 的武器库

### 2.1 库的规模

CANN 的算子库包含了**上千个**预优化好的算子实现，覆盖：

- **基础数学**：Add、Mul、Div、Sqrt、Exp、Log、Pow 等
- **矩阵运算**：MatMul、BatchMatMul、Gemm 等
- **神经网络**：Conv2D、Linear、Softmax、LayerNorm、GELU、ReLU、Dropout 等
- **注意力机制**：FlashAttention、PagedAttention 等融合算子
- **减少运算**：Sum、Mean、Max、Min、ArgMax 等
- **数据操作**：Transpose、Reshape、Concat、Split、Gather、Scatter 等

### 2.2 一个算子背后的复杂性

以 MatMul 为例。一个"高质量"的 MatMul 算子实现需要考虑：

**性能相关**：
- 不同矩阵形状（方阵 vs 瘦长矩阵 vs 宽矮矩阵）的最优分块策略
- Cube 单元的数据格式填充（pad 到 16 的倍数）
- 双缓冲流水线编排（搬运和计算重叠）
- L1/L0 缓存的容量利用最大化

**精度相关**：
- FP16 乘法的中间累加使用 FP32（避免精度丢失）
- 不同输入范围下的数值稳定性
- 与 CPU/GPU 实现的精度对齐

**通用性相关**：
- 支持 FP16、BF16、FP32、INT8 等多种数据类型
- 支持转置操作（A^T × B、A × B^T）
- 支持 Batch 维度（3D/4D 张量）
- 支持动态形状（输入大小运行时才确定）

这就是为什么一个看似简单的数学操作，在工程实现上需要一整个专业团队来优化。

### 2.3 融合算子：性能的杀手锏

除了基础算子，CANN 还提供了一些**融合算子**（Fused Operator），把多个常见的连续操作合并成一个：

| 融合算子 | 等价操作 | 性能收益 |
|---------|---------|---------|
| FlashAttention | MatMul → Scale → Mask → Softmax → MatMul | 减少 HBM 读写 2-4 倍 |
| FusedLayerNorm | Sub → Pow → Mean → Add → Div → Mul → Add | 减少 Vector 操作间的数据搬运 |
| AddLayerNorm | Add（残差）+ LayerNorm | 两步合一，数据不出缓存 |

融合算子是**手工编写**的，需要同时理解数学等价性和硬件特性。CANN 中越来越多的关键路径采用融合算子，这是不断提升性能的重要手段。

## 3. 算子调度机制

### 3.1 算子选择（Dispatch）

当上层调用一个算子时（比如 `aclnnMatMul`），CANN 不是直接执行一段固定代码，而是要做一次**选择**。

因为同一个算子可能有多种实现，分别针对不同的场景优化：

| 输入特征 | 选择的实现 |
|---------|-----------|
| 大方阵 (4096×4096) | 高度优化的 Cube 实现 |
| 极小矩阵 (16×16) | 轻量级 Vector 实现 |
| INT8 数据类型 | INT8 专用 Cube kernel |
| 动态形状 | 通用但稍慢的实现 |

这个选择过程叫 **kernel 选择**或 **dispatch**。选对了，性能最优；选错了，可能差好几倍。

### 3.2 算子编译与缓存

新一代 aclnn 接口的算子调用分两步：

1. **aclnnXxxGetWorkspaceSize**：计算所需工作空间大小，同时进行算子编译
2. **aclnnXxx**：执行算子

第一步的编译结果会被**缓存**。如果后续调用相同形状、类型的算子，直接复用缓存结果，无需重新编译。这对推理场景很友好——推理时输入形状通常固定，第一次调用后编译成本为零。

## 4. 算子兼容性：迁移中的核心挑战

### 4.1 为什么会有算子不支持

当你把一个在 NVIDIA GPU 上训练好的 PyTorch 模型迁移到昇腾时，可能遇到"算子不支持"的报错。原因有几类：

| 原因 | 例子 | 解决方案 |
|------|------|---------|
| CANN 版本太旧 | 新模型的新算子还没加入 | 升级 CANN 版本 |
| 非标准算子 | 模型代码中自定义的特殊操作 | 用支持的算子组合替代，或开发自定义算子 |
| CUDA 专属操作 | 代码中直接调用了 CUDA kernel | 改用 PyTorch 标准 API |
| 数据类型不支持 | 某些算子只支持 FP16 不支持 BF16 | 检查支持矩阵，做精度转换 |

### 4.2 查看算子支持列表

华为维护了一份详细的**算子支持矩阵**，列出了每个 CANN 版本支持哪些算子、支持哪些数据类型、支持哪些形状。这是迁移模型时必须查阅的参考资料。

### 4.3 算子精度差异

即使一个算子在 NPU 和 GPU 上都支持，计算结果也可能存在微小差异。这是因为：

- 不同芯片的浮点运算实现细节不同（运算顺序、舍入规则）
- FP16 的精度只有约 3 位有效小数，累积误差容易放大

对于大多数应用场景，微小精度差异不会影响最终效果。但在某些敏感场景（如强化学习、金融计算），需要特别注意精度对齐。

## 5. 性能分析中的算子视角

当使用 Profiler 分析性能时，算子是最主要的分析粒度。你会看到这样的信息：

```
Top 5 算子（按耗时排序）：
1. MatMul              35%   ← Compute-Bound，正常
2. FlashAttention      20%   ← Compute-Bound，正常
3. TransData           18%   ← 数据格式转换！不正常，需要优化
4. AllReduce           15%   ← 通信开销，需要检查并行策略
5. Softmax              5%   ← Memory-Bound，可接受
```

关键原则：

- **MatMul、Attention 占比高是正常的**——这是 Transformer 的核心计算
- **TransData 占比高是异常的**——说明大量时间花在了数据格式转换上
- **AllReduce 占比高需要评估**——可能通信成为了瓶颈
