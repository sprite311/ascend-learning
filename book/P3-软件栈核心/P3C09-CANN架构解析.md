# P3C09 — CANN 全解析：昇腾软件栈的核心引擎

## 引子：CANN 到底是什么

如果你只能深入了解昇腾的一个技术，那应该就是 CANN。

CANN（Compute Architecture for Neural Networks，异构计算架构）不是一个单一的软件，而是一套由多个组件组成的**核心软件体系**。它做的事情用一句话概括：

**把高层框架描述的 AI 计算任务，翻译、优化、调度、并最终在达芬奇硬件上执行。**

上一章我们知道了昇腾软件栈有六层，CANN 处在中间最关键的位置。本章打开 CANN 的内部，看清它的每一个组件是干什么的、为什么需要它。

---

## 1. CANN 的整体架构

CANN 自身也是分层的，从上到下包含以下主要组件：

```
上层框架调用 (PyTorch/MindSpore)
        │
        ▼
┌───────────────────────────────────────────┐
│              AscendCL (ACL)               │
│           统一编程接口层                    │
├───────────────────────────────────────────┤
│                                           │
│  ┌─────────────┐  ┌───────────────────┐   │
│  │   算子库     │  │    图引擎          │   │
│  │  (Kernels)  │  │ (Graph Engine)    │   │
│  └──────┬──────┘  └────────┬──────────┘   │
│         │                  │              │
│         ▼                  ▼              │
│  ┌────────────────────────────────────┐   │
│  │          Runtime（运行时）           │   │
│  │     任务调度 / 内存管理 / 流管理     │   │
│  └────────────────────────────────────┘   │
│                                           │
└───────────────────────────────────────────┘
        │
        ▼
   NPU Driver → 硬件
```

让我们从上到下逐一解析。

## 2. AscendCL（ACL）：统一的编程入口

### 2.1 它是什么

AscendCL（简称 ACL）是 CANN 暴露给上层的**编程接口**。所有想在昇腾 NPU 上执行计算的程序，最终都要通过 ACL 来下达指令。

ACL 提供了一套 C/C++ 的标准 API，覆盖以下功能：

| API 类别 | 功能 | 示例 |
|---------|------|------|
| 设备管理 | 设置/获取当前使用的 NPU 设备 | `aclrtSetDevice(0)` |
| 内存管理 | 在设备内存（HBM）上分配/释放空间 | `aclrtMalloc()` |
| 数据传输 | 在 CPU 内存和 NPU 内存之间拷贝 | `aclrtMemcpy()` |
| 算子调用 | 执行一个具体的计算操作 | `aclnnMatMul()` |
| 模型推理 | 加载并执行编译好的模型 | `aclmdlExecute()` |
| 流管理 | 管理异步执行的计算流 | `aclrtCreateStream()` |

### 2.2 为什么需要 ACL

**ACL 的存在是为了屏蔽底层复杂性。**

如果你直接和 Runtime 或驱动打交道，需要理解任务队列排布、硬件寄存器映射、中断处理等底层细节。ACL 把这些封装起来，给你一个简洁的接口。

类比：ACL 就像一家餐厅的前台菜单。你只需要说"一份红烧肉"（`aclnnMatMul`），不需要知道厨房里怎么生火、用什么厨具。

### 2.3 ACL vs CUDA

如果你有 CUDA 经验，ACL 的对应关系很直观：

| CUDA | ACL | 说明 |
|------|-----|------|
| `cudaSetDevice()` | `aclrtSetDevice()` | 选择设备 |
| `cudaMalloc()` | `aclrtMalloc()` | 分配设备内存 |
| `cudaMemcpy()` | `aclrtMemcpy()` | 数据拷贝 |
| `cudaStreamCreate()` | `aclrtCreateStream()` | 创建执行流 |
| `cublasSgemm()` | `aclnnMatMul()` | 矩阵乘法 |

概念基本一一对应，学过 CUDA 的人上手 ACL 会比较快。

## 3. 算子库：预制好的"计算零件"

### 3.1 什么是算子

在 AI 计算中，**算子（Operator）** 是最小的可执行计算单元。每种数学操作对应一个算子：

| 算子 | 数学操作 | 对应模型组件 |
|------|---------|-------------|
| MatMul | 矩阵乘法 | Linear 层、QKV 投影 |
| BatchMatMul | 批量矩阵乘法 | Attention Score 计算 |
| Softmax | 指数归一化 | Attention 权重 |
| LayerNorm | 层归一化 | 每层之后的 Norm |
| Add | 逐元素加法 | 残差连接 |
| GELU/SiLU | 激活函数 | FFN 中间 |
| Transpose | 矩阵转置 | 维度变换 |
| Gather/Scatter | 数据索引 | Embedding 查表 |

### 3.2 为什么需要预制算子库

理论上，你可以自己手写每个计算操作的代码。但实际上，一个高性能的矩阵乘法算子需要考虑：

- **分块策略（Tiling）**：怎么切分数据才能最大化缓存利用？
- **数据格式**：怎么把数据排成 Cube 喜欢的 NC1HWC0 格式？
- **流水线编排**：怎么让数据搬运和计算重叠？
- **精度处理**：FP16 乘法中途累加到 FP32 避免溢出？
- **边界处理**：矩阵维度不是 16 的倍数怎么办？

这些优化需要非常深入的硬件知识和大量调试。**华为的底层工程师把这些优化都做好了，封装成算子库**，上层开发者调用即可。

### 3.3 算子库的两代接口

CANN 的算子库经历了一次重要的接口升级：

| 代次 | 接口名 | 特点 |
|------|--------|------|
| 旧一代 | `aclopExecuteV2` | 编译→执行两步（需要先编译算子再调用） |
| 新一代 | **aclnn** 系列 | 编译+执行合一，更简洁高效 |

新的 aclnn 接口（如 `aclnnMatMul`）是目前推荐使用的方式，它内部自动处理了编译、缓存等细节。

### 3.4 当内置算子不够用

CANN 的算子库覆盖了绝大多数常见的 AI 操作。但偶尔你会遇到：

- 某个新模型使用了一个自定义操作，CANN 还没支持
- 某个操作的性能不满足要求，需要针对特定场景深度优化

这时候就需要**自定义算子开发**（P3C12 会讲）。

## 4. 图引擎（Graph Engine）：全局优化的上帝视角

### 4.1 为什么需要图引擎

假设一个 Transformer 层有以下操作序列：

```
MatMul → Add → LayerNorm → MatMul → Softmax → MatMul → Add → LayerNorm → MatMul → GELU → MatMul → Add → LayerNorm
```

如果我们把每个操作独立执行，每次都要：从 HBM 读数据 → 计算 → 写回 HBM → 下一个操作再从 HBM 读。数据在 HBM 和计算单元之间来回搬运，浪费大量时间。

**图引擎的作用就是：站在全局的角度重新安排这些操作，让它们执行得更快。**

### 4.2 图引擎做了什么

图引擎把整个模型当作一张**计算图**（每个算子是节点，数据流动是边），然后做以下优化：

#### 优化一：算子融合（Operator Fusion）

把相邻的多个小操作合并成一个大操作，减少 HBM 读写次数。

```
优化前：
  MatMul → [写HBM] → Add → [写HBM] → LayerNorm → [写HBM]
  (3 次 HBM 读写)

优化后（融合为一个算子）：
  FusedMatMulAddLayerNorm
  (1 次 HBM 读写)
```

性能提升可达 2-3 倍，而且**不改变计算结果**——融合前后的数学等价性由编译器保证。

#### 优化二：常量折叠（Constant Folding）

如果计算图中某些输入在推理时是固定的（比如位置编码 Positional Encoding 的一部分），图引擎会**提前算好结果**，把这个计算节点从推理图中彻底删除。

```
优化前（推理时执行）：
  pos_enc = sin(position * frequency)  // 每次推理都重新算

优化后（编译时已算好）：
  pos_enc = [常量表]  // 直接查表，零计算开销
```

#### 优化三：数据格式自动转换

我们知道 Cube 要求 NC1HWC0 格式的数据。图引擎会在需要的地方**自动插入 TransData 节点**来转换格式，开发者不需要手工处理。

而且，图引擎还会尝试**减少不必要的转换**——如果连续的操作都可以在同一种格式下执行，就不来回转了。

#### 优化四：内存复用

```
优化前：
  Tensor A 用完后释放
  分配新的 Tensor B（和 A 一样大）

优化后：
  Tensor A 用完后不释放
  B 直接复用 A 的内存空间
```

减少内存分配/释放的开销，也降低了 HBM 占用峰值。

### 4.3 ATC 编译器

ATC（Ascend Tensor Compiler）是图引擎对外的工具，它把模型文件（ONNX 格式或其他格式）编译成芯片专用的 .**om 文件**（Offline Model）。

```
模型文件 (.onnx/.pb) → ATC 编译 → 优化后的推理模型 (.om)
```

.om 文件包含了所有上述优化的结果，加载到 NPU 后可以直接高效执行。

## 5. Runtime：任务的调度员

### 5.1 它做什么

Runtime（运行时系统）负责把编译好的计算任务"送上流水线"执行。它的职责包括：

| 功能 | 说明 |
|------|------|
| **任务排队** | 把多个计算任务按顺序排入执行队列 |
| **流管理（Stream）** | 支持多条并行执行流，不同任务可以并行 |
| **内存管理** | 管理 HBM 内存的分配、释放和碎片整理 |
| **同步控制** | 保证有依赖关系的任务按正确顺序执行 |
| **事件机制** | 允许 CPU 端等待 NPU 任务完成 |

### 5.2 流（Stream）的概念

Stream 是 CANN 中实现异步执行的核心概念（和 CUDA Stream 一样）。

```
Stream 1:  [MatMul_A] [MatMul_B] [MatMul_C]
Stream 2:  [Softmax_X] [LayerNorm_Y]
```

同一个 Stream 中的任务按顺序执行（先 A 后 B 后 C）。
不同 Stream 之间的任务可以并行执行（Stream 1 和 Stream 2 同时跑）。

这样可以让 Cube 和 Vector 更充分地被利用——当一个 Stream 的 Cube 任务在等数据时，另一个 Stream 的 Vector 任务可以先执行。

### 5.3 异步执行模型

ACL 的大多数操作是**异步的**：调用 ACL 函数时，CPU 只是向 NPU 发送了一条指令，然后立即返回。实际计算在 NPU 上异步执行。

```python
# 伪代码
aclnnMatMul(...)      # CPU 立即返回，NPU 开始算
aclnnSoftmax(...)     # CPU 继续发指令，NPU 排队执行
aclrtSynchronizeStream()  # CPU 等待 NPU 全部算完
```

这种异步模型允许 CPU 提前准备好很多任务，让 NPU 的执行流水线保持满载——减少"NPU 算完了等 CPU 发下一条指令"的空闲时间。

## 6. 全景视图：一个算子从调用到执行

让我们画一条完整的路径，看看当你在 Python 中执行 `y = torch.matmul(x, w)` 时发生了什么：

```
1. Python 代码:  y = torch.matmul(x, w)
                    │
2. torch_npu:    拦截调用，识别为矩阵乘法
                    │
3. ACL 接口:     调用 aclnnMatMul(x_npu, w_npu, y_npu, stream)
                    │
4. 算子库:       查找 MatMul 的最优算子内核 (kernel)
                 根据输入形状、数据类型选择合适的实现
                    │
5. Runtime:      将 kernel 任务排入 Stream 队列
                 通知 MTE 开始搬运输入数据到 L1/L0
                    │
6. 硬件执行:     MTE 搬运 → Cube 计算 → MTE 写回
                 Scalar 协调整个过程
                    │
7. 结果:         y 的数据出现在 NPU 的 HBM 中
```

从 Python 的一行代码到硬件执行，经过了 7 个步骤、穿越了 4 层软件栈。**每一层都在做对它上层透明的工作。**

## 7. 常见问题与排查思路

| 现象 | 可能原因 | 排查层 |
|------|---------|--------|
| 算子不支持（报 unsupported op） | CANN 版本不含该算子 | 算子库 |
| 性能不佳（Cube 利用率低） | 算子融合没生效 | 图引擎 |
| 精度不对（结果有 NaN） | 数据类型溢出 | 算子库/Runtime |
| 内存溢出（OOM） | Runtime 内存管理 | Runtime |
| CPU 等待时间过长 | 同步过多 | Runtime/ACL |

## 本章小结

| 要点 | 内容 |
|------|------|
| CANN 四大组件 | ACL（接口）+ 算子库（计算）+ 图引擎（优化）+ Runtime（调度） |
| ACL | 统一编程接口，对标 CUDA |
| 算子库 | 预优化的计算函数，体现硬件优化核心能力 |
| 图引擎 | 全局优化：算子融合、常量折叠、格式转换、内存复用 |
| Runtime | 异步执行、流管理、内存管理 |
| 核心认知 | 同一模型在 CANN 优化好和没优化好时性能差距可达 3-10 倍 |
