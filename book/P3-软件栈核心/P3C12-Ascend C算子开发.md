# P3C12 — Ascend C 算子开发：当内置算子不够用时怎么办

## 引子：为什么你可能需要自己写算子

大多数时候，CANN 的内置算子库足以覆盖你的需求。但在以下场景中，你可能需要编写自定义算子：

1. **新模型架构**：一个刚发布的模型使用了全新的计算操作，CANN 还没来得及支持
2. **性能优化**：某个操作用标准算子组合实现性能不够好，需要深度融合
3. **特殊需求**：你的业务有独特的计算逻辑，在标准算子库中找不到对应

本章不是要教你成为算子开发工程师（那需要专门的课程），而是帮你理解 Ascend C 的核心思想，让你知道：

- 自定义算子是怎么工作的
- 它的编程模型要解决什么问题
- 什么时候值得投入精力去开发

---

## 1. 从 TBE 到 Ascend C：编程模型的演进

### 1.1 早期：TBE（Tensor Boost Engine）

昇腾早期的算子开发使用 TBE，基于 Python 的 DSL（领域专用语言）。开发者用 Python 描述计算逻辑，TBE 编译器负责将其翻译成硬件指令。

TBE 的问题：
- 抽象层次太高，开发者难以控制底层细节
- 性能优化空间有限，很多硬件特性无法直接利用
- 和主流 AI 开发者的技能栈（C/C++）脱节

### 1.2 现在：Ascend C

Ascend C 是华为推出的新一代算子编程语言，基于 **C/C++ 扩展**。它的设计目标是：

- 让开发者可以用类 C++ 的语法编写算子
- 同时提供足够的底层控制力来榨取芯片性能
- 保持合理的开发效率（不需要写汇编）

## 2. Ascend C 的核心编程模型

### 2.1 SPMD 并行模型

Ascend C 采用 **SPMD（Single Program Multiple Data）** 编程模型。这个概念很简单：

**你写一份算子代码，它会被复制到芯片上的多个 AI Core 中同时执行。每个 AI Core 处理数据的不同部分。**

类比：一个老师出了一张数学卷子（程序），100 个学生（AI Core）每人分到不同的题目（数据块）同时答。

```
你写的代码（一份）:
  "取出第 block_idx 块数据，计算 matmul，存回结果"

实际执行：
  AI Core 0: 取第 0 块，计算，存回
  AI Core 1: 取第 1 块，计算，存回
  AI Core 2: 取第 2 块，计算，存回
  ...
  AI Core 29: 取第 29 块，计算，存回
```

系统自动为每个 AI Core 分配一个 `block_idx`（块索引），你的代码根据这个索引决定处理哪部分数据。

### 2.2 三级流水线

Ascend C 的核心编程技巧是利用 AI Core 内部的**三级流水线**：

```
阶段 1: CopyIn   (MTE)   ── 把数据从 HBM 搬到片上缓存
阶段 2: Compute  (Cube/Vector) ── 在片上缓存中做计算
阶段 3: CopyOut  (MTE)   ── 把结果从片上缓存写回 HBM
```

这三个阶段**可以流水线并行**：当第 1 批数据在 Compute 阶段时，第 2 批数据可以在 CopyIn 阶段。

```
时间 →
CopyIn:  [批次0] [批次1] [批次2] [批次3]
Compute:         [批次0] [批次1] [批次2]
CopyOut:                [批次0] [批次1]
```

开发者需要显式管理这个流水线——告诉系统哪个阶段可以和哪个阶段重叠。这是性能优化的关键。

### 2.3 Tiling（分块）

一个完整的输入张量通常很大，无法一次性放入 AI Core 的片上缓存（只有几 MB）。所以需要把数据切成小块（Tile），一块一块处理。

```
输入张量 [8192 × 4096]
  ↓ 分块
[256 × 256] × 512 个块
  ↓ 分配
每个 AI Core 处理 512/30 ≈ 17 个块
  ↓ 流水线
每个块按 CopyIn → Compute → CopyOut 流水执行
```

分块策略直接影响性能：
- 块太大 → 放不进片上缓存
- 块太小 → 调度开销太大
- 最优大小取决于算子类型、数据类型和 AI Core 的缓存容量

## 3. Ascend C 代码结构概览

一个典型的 Ascend C 算子代码包含以下部分（简化示意）：

```
Host 端（CPU 上运行）:
├── Tiling 计算    ── 根据输入形状决定如何分块
├── 参数设置       ── 把分块信息传递给设备端
└── 启动 Kernel    ── 触发设备端代码执行

Device 端（NPU 上运行）:
├── Init          ── 初始化缓冲区、设置流水线
├── Process       ── 主循环：遍历所有分块
│   ├── CopyIn    ── 搬运当前块到片上
│   ├── Compute   ── 调用 Cube/Vector 指令计算
│   └── CopyOut   ── 把结果搬回 HBM
└── End           ── 收尾和同步
```

**Host 端**处理"管理逻辑"（在 CPU 上运行）：输入有多大？怎么切块？起多少个 AI Core？

**Device 端**处理"实际计算"（在 NPU 上运行）：每个 AI Core 拿到自己的数据块，执行计算。

## 4. 什么时候值得自己写算子

### 4.1 值得的场景

| 场景 | 原因 |
|------|------|
| 关键路径上的融合优化 | 多个小算子组合的性能远不如融合实现 |
| 全新的计算操作 | CANN 完全没有对应的内置算子 |
| 极致性能要求 | 标准算子的默认实现不够快，需要针对特定形状深度优化 |

### 4.2 不值得的场景

| 场景 | 更好的做法 |
|------|-----------|
| 标准操作但性能不够 | 先检查 CANN 版本是否最新，新版可能已优化 |
| 看起来不支持但可以变通 | 用多个已支持算子组合实现同等效果 |
| 学习单纯想练手 | 建议先学好 CANN 的使用，再考虑算子开发 |

### 4.3 开发成本参考

| 算子类型 | 估算工作量 | 难度 |
|---------|----------|------|
| 简单逐元素操作（如自定义激活函数） | 1-3 天 | 较低 |
| 矩阵操作（如自定义 Attention 变体） | 1-2 周 | 中等 |
| 复杂融合算子（如 FlashAttention 级别） | 1-2 月 | 高 |
