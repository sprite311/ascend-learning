# P4C16 — 模型迁移实战：把 GPU 模型搬到昇腾上

## 引子：迁移的本质是什么

"把一个 GPU 上的 PyTorch 模型迁移到昇腾"这件事听起来是一个任务，但它实际上涉及四个维度的工作：

1. **可以跑**（功能正确）——模型能在 NPU 上运行，结果正确
2. **跑得对**（精度对齐）——NPU 的结果和 GPU 精度一致
3. **跑得快**（性能达标）——性能达到或接近 GPU 水平
4. **跑得稳**（生产可靠）——在生产环境中稳定运行

大多数团队低估了第 2-4 步的工作量。在这里给出一个系统化的迁移流程。

---

## 1. 迁移全流程 Checklist

### 阶段一：准备

- [ ] 确认模型架构和参数规模
- [ ] 查看 CANN 和 torch_npu 的算子支持列表
- [ ] 预判潜在的不支持算子
- [ ] 准备对齐用的测试数据（一小组固定输入）
- [ ] 搭建昇腾开发环境（推荐使用 Docker）

### 阶段二：让它跑起来

- [ ] 修改设备代码（cuda → npu）
- [ ] 修改分布式后端（nccl → hccl）
- [ ] 处理不支持的算子（替代或回退到 CPU）
- [ ] 移除 CUDA 专属操作
- [ ] 确认模型可以完成一次完整的前向传播

### 阶段三：精度对齐

- [ ] 在 CPU 上生成基准输出
- [ ] 比对 NPU 输出和基准的逐层误差
- [ ] 定位精度异常的算子
- [ ] 对敏感算子使用 FP32 或调整实现
- [ ] 验证端到端任务效果（不只看数值差异）

### 阶段四：性能优化

- [ ] Profiling 定位性能热点
- [ ] 检查 TransData（格式转换）占比
- [ ] 检查 CPU-NPU 数据拷贝频率
- [ ] 启用 torch.compile 或图编译优化
- [ ] 调整 Batch Size 提升 Cube 利用率
- [ ] 对比 GPU 性能基线

### 阶段五：生产化

- [ ] 长时间运行稳定性测试
- [ ] 内存泄漏检查
- [ ] 多并发压力测试
- [ ] 监控和告警配置
- [ ] 故障恢复方案

## 2. 算子兼容性处理

遇到不支持的算子时，按以下优先级处理：

### 优先级 1：用标准算子组合替代

```python
# 原始（可能不支持的自定义操作）
output = custom_fused_gelu(x)

# 替代（用标准 PyTorch API）
output = torch.nn.functional.gelu(x)
```

### 优先级 2：回退到 CPU 执行

```python
# 不支持的操作回退到 CPU
x_cpu = x.cpu()
result = unsupported_op(x_cpu)
result = result.npu()  # 结果搬回 NPU
```

注意：CPU 回退会引入数据搬运开销，只适合非核心路径上的操作。

### 优先级 3：开发自定义 Ascend C 算子

如果操作在关键路径上且没有替代方案，考虑用 Ascend C 开发。参考 P3C12。

## 3. 精度对齐方法论

### 3.1 逐层对比法

把模型按层拆开，在 CPU、GPU、NPU 上分别运行，对比每一层输出的差异：

```
Layer 1:  CPU vs NPU 差异 = 1e-6  ✓ 正常
Layer 2:  CPU vs NPU 差异 = 1e-6  ✓ 正常
Layer 3:  CPU vs NPU 差异 = 0.5   ✗ 异常！→ 定位到这一层的算子
```

### 3.2 精度容忍标准

| 场景 | 可接受的差异 |
|------|------------|
| FP32 对比 | < 1e-5（相对误差） |
| FP16 对比 | < 1e-3（相对误差） |
| 端到端任务效果 | BLEU/准确率等指标差异 < 0.5% |

### 3.3 常见精度问题

| 问题 | 原因 | 解决 |
|------|------|------|
| NaN 输出 | FP16 溢出 | 对该层使用 FP32 |
| 结果偏差大 | 累积误差 | 检查 LayerNorm/Softmax 的数值实现 |
| 随机结果不一致 | Dropout 随机种子 | 固定随机种子对比 |

## 4. 性能调优要点

### 4.1 检查清单

| 检查项 | 期望值 | 不达标原因 |
|--------|--------|-----------|
| Cube 利用率 | > 50% | Batch 太小或大量 Vector 操作 |
| TransData 占比 | < 5% | 数据格式频繁转换 |
| HBM 带宽利用 | 接近峰值 | 正常（推理场景本就是带宽瓶颈） |
| CPU-NPU 拷贝 | 极少 | 数据预处理未在 NPU 上完成 |
| Stream 空闲间隙 | 极少 | 同步点太多或任务不饱和 |

### 4.2 性能优化三板斧

1. **增大 Batch Size**：提高算术强度，让 Cube 更忙
2. **减少数据搬运**：确保数据全程在 NPU 上，避免 CPU-NPU 来回拷贝
3. **启用编译优化**：使用 `torch.compile(backend='npu')` 或 ATC 编译

## 5. 迁移常见坑

| 序号 | 坑 | 预防 |
|------|----|----|
| 1 | 版本不匹配导致环境崩溃 | 用 Docker，查版本矩阵 |
| 2 | `model.half()` 转 FP16 后 NaN | 用 AMP 而不是手动转精度 |
| 3 | DataLoader 的 `pin_memory=True` 报错 | 昇腾不支持 pin_memory，设为 False |
| 4 | 多卡训练时 HCCL 初始化超时 | 检查 npu-smi、网络连通性 |
| 5 | 性能测试时没做 Warmup | 前几次迭代包含编译开销，不能计入 |
