# P5C21 — 推理优化技术全景：量化、KV Cache、算子融合

## 引子

前面的章节介绍了推理引擎的框架设计。本章聚焦于推理引擎内部使用的**核心优化技术**——它们是推理性能的决定性因素。

---

## 1. 量化：用精度换速度

### 1.1 为什么量化有效

回顾 P2C06 的分析：LLM 推理的 Decode 阶段是 Memory-Bound。性能瓶颈在于"搬运权重的速度太慢"。

量化的思路极其直接：**把权重从 FP16（2 字节/参数）压缩到 INT8（1 字节）或 INT4（0.5 字节），搬运体积立刻减半或减到四分之一。**

而且，Cube 单元处理 INT8 的吞吐量是 FP16 的 4 倍。所以量化同时带来两重加速：搬运快 + 计算快。

### 1.2 量化方式

| 方式 | 说明 | 精度影响 | 性能收益 |
|------|------|---------|---------|
| **W8A8** | 权重 INT8，激活 INT8 | 小 | ~2x |
| **W8A16** | 权重 INT8，激活 FP16 | 更小 | ~1.5x |
| **W4A16** | 权重 INT4，激活 FP16 | 中 | ~2x |
| **GPTQ** | 逐层量化校准 | 小 | ~2x |
| **AWQ** | 保护重要权重通道 | 小 | ~2x |

### 1.3 在昇腾上量化

CANN 提供了量化工具 **AMCT**（Ascend Model Compression Toolkit）。它支持训练后量化（PTQ）和量化感知训练（QAT）。

MindIE 和 vLLM-Ascend 都支持加载量化后的模型进行推理。

## 2. KV Cache 管理

### 2.1 KV Cache 的挑战

KV Cache 是 LLM 推理最大的内存消耗项（通常占总显存的 30-60%）。管理不好会导致：
- 并发用户数受限（KV Cache 占满了显存）
- 内存碎片化（不同请求的 Cache 大小不同）

### 2.2 PagedAttention 的内存管理

PagedAttention 把 KV Cache 分成固定大小的"块"（Block），用一个"块表"（Block Table）维护每个请求的 KV Cache 在物理内存中的位置。

优势：
- **零碎片**：所有块大小相同，不存在内存碎片
- **按需分配**：序列增长时才分配新块
- **共享复用**：Beam Search 等场景下，不同候选序列可以共享前缀的 KV Cache

### 2.3 KV Cache 量化

除了权重量化，KV Cache 本身也可以量化（FP16 → INT8/FP8），进一步减少内存占用：

```
FP16 KV Cache: 每个请求 2 GB
INT8 KV Cache: 每个请求 1 GB → 同样显存能服务双倍并发
```

## 3. FlashAttention

### 3.1 原始 Attention 的问题

标准 Attention 的计算过程中，Score 矩阵（Q × K^T）的大小是 [seq_len × seq_len]。当序列长度为 4096 时，这是一个 4096×4096 的矩阵，FP16 下占 32 MB。

完整流程需要多次读写这个大矩阵到 HBM，产生大量的数据搬运。

### 3.2 FlashAttention 的解决方案

核心思想：**不在 HBM 中存储完整的 Score 矩阵，而是按块（Block）计算，每块的中间结果都留在 SRAM（片上缓存）中。**

通过一种叫 **Online Softmax** 的数学技巧，可以在不看到全部 Score 的情况下正确计算 Softmax——这使得逐块计算成为可能。

```
标准 Attention:
  HBM ←→ Score(全量) ←→ Softmax(全量) ←→ Output
  多次 HBM 读写

FlashAttention:
  逐块在 SRAM 中: Score(块) → Softmax(块) → 部分 Output
  极少 HBM 读写
```

在昇腾上，CANN 提供了融合的 FlashAttention 算子实现，推理引擎默认使用。

## 4. Continuous Batching 详解

### 4.1 静态 Batching 的问题

```
Batch = [请求A(输出100Token), 请求B(输出10Token)]

时间 →
A: [════════════════════════════════════]  100 tokens
B: [════]·······························   10 tokens, 然后空等

问题：B 完成后 NPU 对 B 的那部分计算资源就浪费了
```

### 4.2 Continuous Batching

```
时间 →
A: [══════════════════════════════════════]  100 tokens
B: [════]
C:      [════════════]                       B 走后 C 顶上
D:                   [══════]                C 走后 D 顶上
```

NPU 始终满载，没有空闲浪费。

### 4.3 Iteration-Level Scheduling

在 Continuous Batching 中，调度最小单位是一次 **Iteration**（一次 Decode 步骤，即生成一个 Token）。每次 Iteration 结束后，调度器检查：
- 哪些请求已完成？→ 释放
- 有没有等待的新请求？→ 加入
- KV Cache 够不够？→ 必要时驱逐（Preempt）低优先级请求

## 5. 优化技术的协同

这些技术不是独立使用的，而是协同工作：

```
┌─────────────────────────────────────┐
│         Continuous Batching          │  ← 最大化 NPU 利用率
│  ┌─────────────────────────────┐    │
│  │    PagedAttention            │    │  ← 最大化 KV Cache 效率
│  │  ┌─────────────────────┐    │    │
│  │  │  FlashAttention      │    │    │  ← 最小化 HBM 读写
│  │  │  ┌───────────────┐   │    │    │
│  │  │  │  量化 (INT8)   │   │    │    │  ← 最小化搬运体积
│  │  │  └───────────────┘   │    │    │
│  │  └─────────────────────┘    │    │
│  └─────────────────────────────┘    │
└─────────────────────────────────────┘
```

叠加使用后，推理性能可以达到原始 PyTorch `model.generate()` 的 **10-30 倍**。
