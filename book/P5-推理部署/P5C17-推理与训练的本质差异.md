# P5C17 — 推理 vs 训练：为什么要区别对待

## 引子

很多刚入行的工程师以为："部署推理不就是把训练好的模型拿来跑一下 forward 吗？"

如果真这么简单，就不会有 TensorRT、vLLM、MindIE 这些专门的推理引擎了。推理和训练虽然共享同一个模型架构，但在计算特性、内存模型、优化策略上有根本性差异。

---

## 1. 核心差异

### 1.1 计算方向

| | 训练 | 推理 |
|---|------|------|
| 前向传播 | ✓ | ✓ |
| 反向传播 | ✓ | ✗ |
| 梯度计算 | ✓ | ✗ |
| 参数更新 | ✓ | ✗ |

推理只需做前向传播，计算量约为训练的 **1/3**（没有反向传播和梯度更新）。

### 1.2 内存使用模式

训练时，为了计算反向传播的梯度，需要**保存所有中间层的激活值**——这是显存的大头。

推理时不需要保存激活值，但大模型推理引入了一个新的内存消耗大户——**KV Cache**。

### 1.3 计算模式

训练通常处理**固定长度的大 Batch**，算术强度高，是 Compute-Bound。

大模型推理的 Decode 阶段是**逐 Token 生成**，每次只算一个 Token，算术强度极低，是 Memory-Bound。

---

## 2. LLM 推理的两个阶段

大语言模型的推理可以分为两个截然不同的阶段：

### 2.1 Prefill（填充阶段）

处理用户输入的完整 prompt。所有输入 Token 一次性并行计算。

- **特点**：计算量大、一次完成、类似训练的前向传播
- **瓶颈**：通常是 Compute-Bound
- **衡量指标**：TTFT（Time To First Token，首 Token 延迟）

### 2.2 Decode（生成阶段）

逐 Token 生成回复。每次生成一个 Token，以之前所有 Token 的信息为基础。

- **特点**：每次只处理一个 Token，但需要读取全部权重
- **瓶颈**：严重的 Memory-Bound
- **衡量指标**：Token/s（每秒生成 Token 数）

```
用户输入: "请介绍一下昇腾芯片"（10 个 Token）
         ↓
[Prefill] 一次处理 10 个 Token → 得到第 11 个 Token
         ↓
[Decode]  用第 1-11 Token 算 → 第 12 个 Token
[Decode]  用第 1-12 Token 算 → 第 13 个 Token
[Decode]  用第 1-13 Token 算 → 第 14 个 Token
...持续到生成结束标记
```

---

## 3. KV Cache：推理的内存大户

### 3.1 为什么需要 KV Cache

在 Decode 阶段，每生成一个新 Token，Attention 计算需要用到**所有历史 Token 的 Key 和 Value 向量**。

如果每次都重新计算所有历史 Token 的 KV，计算量随序列长度线性增长——太浪费。

KV Cache 的思路是：**把已经计算过的 Key/Value 缓存起来，每次只计算新 Token 的 KV**。

### 3.2 KV Cache 的内存占用

一个 7B 模型、序列长度 4096：

```
KV Cache = 层数 × 2(K和V) × 序列长度 × 头数 × 维度 × 字节数
         = 32 × 2 × 4096 × 32 × 128 × 2(FP16)
         ≈ 2 GB（每个请求）
```

如果同时服务 16 个用户，KV Cache 就需要 **32 GB**，已经占了一张 910B（64 GB）的一半空间。

KV Cache 的管理效率直接决定了一张卡能同时服务多少并发用户。

---

## 4. 推理的核心优化方向

| 优化方向 | 解决什么问题 | 技术 |
|---------|------------|------|
| 减少搬运量 | Memory-Bound | 量化（INT8/INT4） |
| 提高搬运效率 | 带宽利用 | 算子融合、FlashAttention |
| 减少重复计算 | 计算浪费 | KV Cache |
| 提高并发 | 吞吐量 | Continuous Batching |
| 降低内存浪费 | KV Cache 碎片 | PagedAttention |

这些技术将在后续章节详细展开。

## 本章小结

| 要点 | 内容 |
|------|------|
| 训练 vs 推理本质 | 训练是 Compute-Bound，推理是 Memory-Bound |
| 两个推理阶段 | Prefill（并行处理 prompt）和 Decode（逐 Token 生成） |
| KV Cache | 缓存历史 KV 避免重复计算，但消耗大量内存 |
| 优化方向 | 量化、融合、KV Cache、动态批处理 |
