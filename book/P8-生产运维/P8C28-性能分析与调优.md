# P8C28 — 性能分析与调优：用科学方法找到瓶颈

## 1. 核心原则：不分析，不调优

性能优化中最大的忌讳是"凭直觉调参"——改这改那，按下葫芦浮起瓢。正确的做法是：**先用 Profiler 定位瓶颈，再针对性优化。**

**阿姆达尔定律**（Amdahl's Law）是铁律：如果一个部分只占总时间的 10%，即使把它优化到零，总时间也最多减少 10%。所以**永远只对占比最大的瓶颈下刀**。

---

## 2. 性能瓶颈分类

| 类型 | 特征 | 典型原因 |
|------|------|---------|
| **Compute-Bound** | Cube 利用率高，HBM 带宽低 | Batch 大、矩阵运算密集 |
| **Memory-Bound** | Cube 利用率低，HBM 带宽打满 | 推理 Decode、Batch 太小 |
| **Communication-Bound** | AllReduce 耗时占比高 | 多卡通信成为瓶颈 |
| **Host-Bound** | Device 端有大量空闲间隙 | CPU 数据预处理太慢或同步过多 |

---

## 3. 调优流程

```
步骤 1: Profiling 采集数据
步骤 2: 看端到端延迟组成（Host vs Device）
步骤 3: 定位 Device 端 Top-5 算子
步骤 4: 判断瓶颈类型
步骤 5: 针对性优化
    ├── Memory-Bound → 量化/融合/增大Batch
    ├── Compute-Bound → 混合精度/算子优化
    ├── Communication-Bound → 调整并行策略/减少通信
    └── Host-Bound → 异步数据加载/减少同步点
步骤 6: 重新 Profiling 验证效果
```

---

## 4. 常见优化手段速查

| 手段 | 适用场景 | 预期收益 |
|------|---------|---------|
| 增大 Batch Size | Memory-Bound | 吞吐提升 2-4x |
| INT8 量化 | Memory-Bound | 速度提升 1.5-2x |
| 算子融合 | 大量小算子 | 速度提升 1.5-3x |
| torch.compile | 碎片化操作 | 速度提升 1.2-2x |
| 减少同步点 | Host-Bound | 减少空闲间隙 |
| 调整 TP/PP 比例 | Communication-Bound | 减少通信开销 |
