# P9C34 — 场景四：端侧与边缘推理部署

## 引子

不是所有 AI 应用都在数据中心运行。很多场景需要在**边缘设备**上就地推理——摄像头实时检测行人、工厂产线实时质检、无人机自主避障。这些场景的特点是：低功耗、低延迟、无网络依赖。

---

## 1. 边缘推理的特点

| 维度 | 云端推理 | 边缘推理 |
|------|---------|---------|
| 硬件 | Atlas 800（910B） | Atlas 200（310/310P） |
| 功耗 | 数百瓦 | 数瓦到数十瓦 |
| 算力 | 数百 TFLOP/s | 数十 TOPS |
| 内存 | 64 GB HBM | 数 GB DDR/LPDDR |
| 网络 | 高带宽 | 可能无网络 |
| 模型大小 | 数 GB - 数百 GB | 数 MB - 数百 MB |

---

## 2. 模型优化技术

要把模型塞进边缘设备，通常需要多种压缩技术组合使用：

### 2.1 量化

INT8 甚至 INT4 量化，把模型大小压缩 2-4 倍，同时利用 310 的 INT8 高效推理能力。

### 2.2 剪枝

去掉模型中对精度贡献小的参数（权重接近零的连接），减少计算量和模型大小。

### 2.3 知识蒸馏

训练一个"小学生模型"来模仿"大教授模型"的行为。小模型参数少、推理快，但效果接近大模型。

### 2.4 模型量化压缩流程

```
大模型 (FP32, 200MB)
  ├── 剪枝 → 150MB
  ├── 蒸馏 → 50MB（小模型）
  └── 量化 → 25MB（INT8）
        ↓
  ATC 编译 → .om
        ↓
  部署到 Atlas 200
```

## 3. MindSpore Lite

MindSpore Lite 是 MindSpore 的轻量版，专为端侧和边缘推理设计：

- **极小运行时**：最小 1MB 以下
- **多平台**：支持昇腾 NPU、手机 CPU/GPU、嵌入式 Linux
- **模型格式**：使用 .ms 格式（从 ONNX 或 MindIR 转换而来）

---

## 4. 端云协同

实际部署中，很多场景采用"端侧初筛 + 云端精处理"的协同模式：

```
摄像头 → [边缘: 目标检测] → 发现异常？
                                ├── 否 → 不处理
                                └── 是 → 截图上传云端
                                         → [云端: 大模型精细分析]
                                         → 返回分析结果
```

边缘设备负责高频低精度的初筛，只在发现值得关注的情况时才求助云端。这样既控制了网络带宽和云端计算成本，又保证了关键事件不遗漏。
