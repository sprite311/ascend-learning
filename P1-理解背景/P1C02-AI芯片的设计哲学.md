# P1C02 — AI 芯片的设计哲学：从 GPU 到 NPU 的演进逻辑

## 引子：为什么会有不同的 AI 芯片路线？

上一章我们知道了，AI 计算的本质是大规模矩阵乘法，CPU 搞不定，需要专用芯片。但如果你关注芯片行业，会发现一个有趣的现象：

- NVIDIA 选择了 GPU 路线（CUDA 生态）
- Google 造了 TPU（张量处理单元）
- 华为设计了昇腾 NPU（达芬奇架构）

如果问题这么清晰（就是矩阵乘法嘛），为什么这些公司的答案不一样？它们之间是什么关系？各自做了怎样的取舍？

理解这些不同路线背后的设计哲学，是理解昇腾定位的前提。

---

## 1. GPU 路线：从"意外发现"到主导地位

### 1.1 GPU 的起源

GPU 最初和 AI 没有任何关系。它是为了**实时渲染 3D 图形**而设计的——游戏中每一帧画面的每个像素都需要独立计算颜色和光影，这天然需要大量并行计算。

所以 GPU 从诞生之日起就走了和 CPU 相反的路：用大量简单核心做并行计算。

2006 年，NVIDIA 做了一个关键决策：推出 **CUDA**（Compute Unified Device Architecture）。CUDA 不是硬件，而是一套编程模型和工具链，它让程序员可以用 C/C++ 语言来驱动 GPU 做**通用计算**（而不仅仅是图形渲染）。

### 1.2 "偶然"进入 AI

2012 年，Alex Krizhevsky 用两块 NVIDIA GTX 580 GPU 训练了一个叫 AlexNet 的神经网络，在 ImageNet 图像识别竞赛中大幅刷新纪录。这个结果震动了整个学术界。

GPU 加速 AI 不是 NVIDIA 主动规划的——是学术界发现 GPU 的并行架构恰好适合神经网络的矩阵运算。但 NVIDIA 迅速抓住了这个机遇，投入大量资源优化 CUDA 在 AI 场景下的表现。

### 1.3 NVIDIA 的护城河：不只是硬件

到今天，NVIDIA 的优势不仅仅是 GPU 硬件好，而是建立了一个极为庞大的**软件生态**：

| 生态组件 | 作用 |
|---------|------|
| CUDA | 底层编程模型 |
| cuBLAS | 高度优化的矩阵运算库 |
| cuDNN | 深度学习算子库 |
| TensorRT | 推理优化引擎 |
| NCCL | 多卡通信库 |

几乎所有主流 AI 框架（PyTorch、TensorFlow）都是围绕 CUDA 生态构建的。这意味着全世界的 AI 代码默认就是跑在 NVIDIA GPU 上的。

**这就是昇腾面临的核心竞争问题：不是硬件不够好，而是生态迁移的成本巨大。** 后面我们会看到昇腾如何应对这个挑战。

## 2. TPU 路线：Google 的另一种思路

### 2.1 为什么 Google 要自己造芯片？

Google 是全球最大的 AI 用户之一（搜索排名、翻译、推荐系统、Gmail 智能回复……）。2013 年前后，Google 内部做了一个预测：如果所有用户每天对 Google 说三分钟语音（语音搜索），按当时的 AI 模型算力需求，Google 需要**把数据中心的服务器数量翻倍**。

这个成本是不可接受的。于是 Google 决定自己造芯片。

### 2.2 TPU 的设计哲学：极致专用

TPU（Tensor Processing Unit）的设计思路比 GPU 更激进：

- **不做通用计算**：TPU 几乎不保留任何通用计算能力，全部面积给矩阵运算
- **脉动阵列（Systolic Array）**：TPU v1 的核心是一个 256×256 的脉动阵列，数据像波浪一样在阵列中流动，每经过一个节点就完成一次乘加运算
- **绑定 TensorFlow**：TPU 最初只能通过 TensorFlow 使用，后来扩展到 JAX

### 2.3 TPU 的取舍

TPU 追求极致的矩阵计算吞吐量，但代价是：
- **通用性极差**：不支持图形渲染，甚至很多非标准 AI 操作也不支持
- **生态封闭**：主要通过 Google Cloud 提供，不单独卖芯片
- **编程灵活性低**：开发者必须使用特定的框架和 API

## 3. 昇腾 NPU 路线：华为的设计选择

### 3.1 华为为什么要做 AI 芯片？

华为做昇腾的背景有两个层面：

**技术层面**：华为在通信设备（基站、路由器）中积累了大量的芯片设计经验（海思），而 AI 正在渗透到通信网络的每一个环节（智能网络调优、基站节能等）。

**战略层面**：在全球 AI 计算被 NVIDIA 高度垄断的格局下，拥有自主可控的 AI 计算能力是国家级的战略需求。

### 3.2 达芬奇架构：昇腾的核心设计

2018 年，华为正式发布了昇腾 AI 芯片，核心架构被命名为"达芬奇架构"（Da Vinci Architecture）。

达芬奇架构的设计哲学可以概括为：**在矩阵计算的专用性和计算任务的通用性之间寻找平衡**。

它的核心计算单元叫 **AI Core**，每个 AI Core 内部有三种计算单元：

| 计算单元 | 职责 | 类比 |
|---------|------|------|
| **Cube**（张量计算单元） | 做大矩阵乘法 | 工厂里的大型冲压机 |
| **Vector**（向量计算单元） | 做激活函数、归一化等向量运算 | 精密加工车间 |
| **Scalar**（标量计算单元） | 做流程控制、地址计算 | 车间主任（调度指挥） |

为什么要三种单元同时存在？

因为一个完整的 AI 模型不只有矩阵乘法。以 Transformer 的一个层为例：
1. 先做 QKV 矩阵乘法 → **Cube 负责**
2. 然后做 Softmax（指数运算+归一化）→ **Vector 负责**
3. 再做 Attention 输出的矩阵乘法 → **Cube 负责**
4. 最后做 LayerNorm → **Vector 负责**

如果只有 Cube（像 TPU 那样极致专用），遇到 Softmax 和 LayerNorm 就得把数据搬回 CPU 处理，搬运的开销会吃掉所有性能增益。

如果只有 Vector（像 GPU 那样通用），矩阵乘法的效率就上不去。

达芬奇架构的选择是：**让三种单元各司其职，协同流水线工作**。Cube 做完矩阵乘法，结果不出芯片，直接在片上交给 Vector 做后处理。这就是"三驾马车"的设计逻辑。

### 3.3 关键设计：Cube 的 16×16×16 矩阵计算

Cube 单元的核心能力是：**在一个时钟周期内，完成一个 16×16 矩阵乘以一个 16×16 矩阵的运算**。

为什么是 16×16？这是在以下因素之间做的平衡：
- 阵列越大，单次计算吞吐越高
- 但阵列越大，数据填充（把数据排好送进去）的开销越大
- 16×16 是在当前工艺下，效率和利用率的一个甜蜜点

一次 16×16×16 的运算 = 16×16×16 = 4096 次乘加操作。如果一个芯片有 32 个 AI Core，每个 AI Core 有 1 个 Cube，那一个时钟周期就是 32 × 4096 = 131,072 次乘加。时钟频率假设 1.5 GHz，理论算力约为 **393 TFLOP/s**（FP16）。

## 4. 三条路线的对比与启示

### 4.1 横向对比

| 维度 | NVIDIA GPU | Google TPU | 华为昇腾 NPU |
|------|-----------|------------|-------------|
| 核心设计 | CUDA Core + Tensor Core | 脉动阵列 | Cube + Vector + Scalar |
| 通用性 | 高（还能做图形、科学计算） | 低（几乎只能做 AI） | 中（专注 AI但保留灵活性） |
| 生态成熟度 | 极高（CUDA 生态 15+ 年） | 中（绑定 Google Cloud） | 成长中（CANN 生态快速发展） |
| 获取方式 | 购买显卡 | 租用 Google Cloud | 购买 Atlas 设备/华为云 |
| 编程接口 | CUDA / cuDNN | XLA / JAX | CANN / ACL / Ascend C |
| 互联技术 | NVLink / InfiniBand | ICI（芯片间互联） | HCCS / RoCE |

### 4.2 核心启示

**没有"最好"的 AI 芯片，只有"最适合特定场景"的设计取舍。**

1. 如果你追求**最大的生态兼容性**和**开箱即用**——NVIDIA GPU 目前无可替代
2. 如果你是**Google Cloud 用户**追求**极致吞吐**——TPU 是不错的选择
3. 如果你的场景需要**自主可控的全栈 AI 能力**，或者在**华为生态内**——昇腾是核心选择

理解了每条路线的设计逻辑和取舍，你就不会被"谁更强"这种简单化的问题误导。真正重要的是：**理解每个设计决策背后的"为什么"，然后根据你的实际需求选择。**

## 5. 一个更深层的洞察：生态比芯片更重要

在芯片行业有一个说法：**"芯片的竞争力，30% 在硬件，70% 在软件生态。"**

NVIDIA 的 GPU 在纯硬件性能上，并不总是碾压所有竞争者。但它的 CUDA 生态——十几年积累的编程模型、优化库、开发者社区、学术论文中默认使用的框架——构成了一道几乎无法逾越的护城河。

这就是为什么昇腾要构建一个**完整的生态系统**（而不是只造一颗芯片）：

| 层次 | NVIDIA 对标 | 昇腾对应 |
|------|-----------|---------|
| 编程模型 | CUDA | CANN / Ascend C |
| 算子库 | cuBLAS / cuDNN | CANN 算子库 |
| 图编译器 | TensorRT | ATC / 图引擎 |
| 框架适配 | 原生支持 PyTorch | torch_npu 插件 |
| 通信库 | NCCL | HCCL |
| 推理引擎 | TensorRT-LLM / vLLM | MindIE / vLLM-Ascend |
| 自有框架 | — | MindSpore |
