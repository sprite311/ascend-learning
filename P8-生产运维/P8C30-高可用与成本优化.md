# P8C30 — 高可用架构与成本优化：AI 服务的工程化

## 1. 高可用设计

### 1.1 为什么 AI 服务容易挂

AI 推理服务比传统 Web 服务更脆弱：
- **资源密集**：一个请求可能占用一张卡的全部算力好几秒
- **显存敏感**：KV Cache 突增可能导致 OOM
- **硬件故障**：NPU 硬件故障率比普通服务器更高

### 1.2 关键设计要素

| 要素 | 做法 |
|------|------|
| **冗余部署** | 至少 2 个推理实例，负载均衡 |
| **健康检查** | 定期发送探测请求，检查推理服务是否存活 |
| **自动重启** | OOM 或卡死后自动重启进程/容器 |
| **限流** | 设置最大并发数，防止过载 |
| **优雅降级** | 极端负载下返回简化回复而不是报错 |
| **Checkpoint** | 训练任务定期保存状态，故障后从 checkpoint 恢复 |

### 1.3 监控告警

建议监控的关键指标：

| 指标 | 告警阈值 |
|------|---------|
| NPU 温度 | > 85°C |
| HBM 使用率 | > 90% |
| 推理延迟 P99 | > SLA 阈值 |
| 请求错误率 | > 1% |
| NPU 利用率 | 持续 < 10%（资源浪费） |

## 2. 成本优化

### 2.1 AI 服务的成本结构

| 成本项 | 占比 | 优化方向 |
|--------|------|---------|
| **硬件（NPU 卡）** | ~60% | 提高利用率、量化减少卡数 |
| **电力与散热** | ~20% | 降低功耗、优化调度 |
| **网络与存储** | ~10% | 合理规划 |
| **运维人力** | ~10% | 自动化 |

### 2.2 提高利用率的方法

| 方法 | 效果 |
|------|------|
| **量化**（FP16 → INT8） | 同等性能减少一半的卡 |
| **Continuous Batching** | 提高吞吐量，同样的卡服务更多用户 |
| **分时复用** | 白天跑推理服务，晚上跑训练任务 |
| **弹性伸缩** | 流量低时减少实例数 |
| **模型蒸馏** | 用小模型替代大模型，性能差距不大但成本大幅降低 |

### 2.3 TCO（总体拥有成本）估算

简化的 TCO 公式：

```
年成本 ≈ 硬件采购 / 折旧年数 × (1 + 运维系数)
       + 电力单价 × 功耗(kW) × 8760(小时) × PUE

示例（8 卡 Atlas 800）：
  硬件：~80 万 / 3 年 = 26.7 万/年
  电力：310W×8 = 2.48kW × 1.0/度 × 8760 × 1.4(PUE) ≈ 3 万/年
  运维：~5 万/年
  总计：~35 万/年
```
