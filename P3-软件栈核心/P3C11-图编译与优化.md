# P3C11 — 图编译器：如何让模型"无痛"提速

## 引子：不改代码，性能翻倍？

有一个常见的场景让初次接触昇腾的开发者很困惑：

一个模型在 PyTorch 中用 `model.generate()` 直接跑，推理速度是 X。然后用 ATC 编译器转换成 .om 格式后再跑，速度变成了 3X——**快了三倍，但代码和模型权重一个字都没改过**。

这三倍的加速是从哪来的？

答案就是**图编译器**。它在你看不见的地方，对模型的计算流程做了大量的重排和优化。本章让你彻底理解它做了什么、为什么能做到、以及什么时候它可能做不好。

---

## 1. 什么是计算图

### 1.1 从代码到图

当你在 PyTorch 中定义和运行一个模型时，本质上是在执行一系列有先后顺序的计算操作。这些操作和它们之间的数据依赖关系，可以表示为一张**有向无环图（DAG）**：

```
输入 X
  │
  ├──→ [MatMul: W_q] ──→ Q ─────────┐
  ├──→ [MatMul: W_k] ──→ K ──→ [K^T] ──→ [MatMul: Q×K^T] ──→ Score
  └──→ [MatMul: W_v] ──→ V              │                       │
                                         └───────────────────────┤
                                                                 │
                                              [Scale: /√d] ←─────┘
                                                   │
                                              [Softmax] ──→ Weights
                                                                 │
                                              [MatMul: W×V] ←───┘
                                                   │
                                              [Add: 残差]
                                                   │
                                              [LayerNorm]
                                                   │
                                                 输出
```

每个方括号是一个**节点**（算子），箭头是**边**（数据流向）。这就是计算图。

### 1.2 为什么"图"比"代码"更适合优化

人写代码时是**顺序思维**——一行一行写，每行做一件事。但很多优化需要**全局视角**——知道前后文的所有操作，才能发现优化机会。

这就是图的优势：**编译器可以一眼看到整个计算流程，发现人看不到的优化空间。**

## 2. 五大图优化技术

### 2.1 算子融合（Operator Fusion）

**这是最重要的优化，也是收益最大的。**

问题回顾：每个算子单独执行时，中间结果要写回 HBM，下一个算子再从 HBM 读。频繁的 HBM 读写是 Memory-Bound 操作的性能杀手。

融合的做法：把多个相邻算子合并成一个"大算子"，中间结果留在片上缓存（L0/L1）中传递，不写回 HBM。

**实际例子——Attention 融合**：

```
融合前（5 次 HBM 读写）：
  Score = MatMul(Q, K^T)   → 写回 HBM
  Score = Score / √d        → 读出，计算，写回 HBM
  Score = Mask(Score)       → 读出，计算，写回 HBM
  Weights = Softmax(Score)  → 读出，计算，写回 HBM
  Output = MatMul(Weights, V) → 写回 HBM

融合后（FlashAttention，1 次 HBM 读写）：
  Output = FlashAttention(Q, K, V, mask)
  所有中间结果在 SRAM 中计算完成
```

对于长序列（如 4096 Token），这个优化可以节省 **TB 级别的 HBM 读写量**，速度提升 2-4 倍。

### 2.2 常量折叠（Constant Folding）

如果计算图中某些节点的输入在推理时永远不变（比如位置编码的一些预计算值、固定的缩放系数），编译器会**在编译时就把这些节点算好**，运行时直接使用结果。

```
优化前（运行时每次都算）：
  scale = 1.0 / sqrt(64.0)    ← 每次推理都做一次 sqrt 和除法
  score = matmul(Q, K) * scale

优化后（编译时已算好）：
  score = matmul(Q, K) * 0.125  ← 直接乘一个常数
```

看起来微不足道，但在一个有几十层的模型中，这类操作累计起来也能节省可观的时间。

### 2.3 数据布局优化（Layout Transformation）

我们在 P2C04 中讲过，Cube 要求 NC1HWC0 格式的数据。如果不转换格式，就无法使用 Cube 做高效矩阵运算。

图编译器做两件事：

1. **自动插入转换节点**：在需要的地方（比如从 CPU 送往 Cube 之前）自动加上 TransData
2. **优化转换位置**：尽可能减少转换次数

```
优化前：
  [NCHW] → MatMul → [NCHW] → Add → [NCHW] → MatMul → [NCHW]
  每次 MatMul 前后都要转格式 = 4 次 TransData

优化后：
  [NCHW] → TransData → [NC1HWC0] → MatMul → Add → MatMul → TransData → [NCHW]
  只在首尾转换 = 2 次 TransData
```

### 2.4 内存复用（Memory Reuse）

在计算图中，很多中间张量的生命周期很短——用完就不再需要了。编译器会分析每个张量的**存活区间**，让不同时存活的张量**共用同一块内存**。

```
Tensor A: 活跃于 [t0, t3]
Tensor B: 活跃于 [t4, t7]

普通分配：A 占 100MB，B 占 100MB → 总共 200MB
复用后：  B 复用 A 的空间 → 总共 100MB
```

这在大模型中效果尤为明显——减少 HBM 峰值占用，可能让一个"放不下"的模型变得"刚好放下"。

### 2.5 死节点消除（Dead Node Elimination）

如果某个节点的输出不被任何后续节点使用（可能是模型导出时残留的调试代码或无用分支），编译器会直接删掉它。

```
优化前：
  X → [MatMul] → Y     ← Y 被后续使用
  X → [Debug_Log] → Z  ← Z 没有被任何节点使用

优化后：
  X → [MatMul] → Y     ← 保留
  (Debug_Log 被删除)    ← 节省计算和内存
```

## 3. ATC 编译器的工作流程

ATC（Ascend Tensor Compiler）是昇腾的模型编译工具，它接收一个模型文件（通常是 ONNX 格式），输出一个优化后的 .om 文件。

完整的编译流程：

```
输入: model.onnx
      │
  ┌───▼───────────────────┐
  │   1. 模型解析           │  解读 ONNX 格式，构建内部计算图
  ├───────────────────────┤
  │   2. 图级优化           │  算子融合、常量折叠、死节点消除
  ├───────────────────────┤
  │   3. 算子选择           │  为每个节点选择最优的 kernel 实现
  ├───────────────────────┤
  │   4. 数据布局优化       │  决定每个张量的最优存储格式
  ├───────────────────────┤
  │   5. 内存规划           │  分配和复用 HBM 空间
  ├───────────────────────┤
  │   6. 代码生成           │  生成最终的硬件指令序列
  └───────────────────────┘
      │
输出: model.om
```

### 3.1 静态图 vs 动态图

ATC 编译要求模型的**形状在编译时确定**（静态图）。这意味着：

- 输入的 batch size、序列长度等维度在编译时就要指定
- 如果运行时输入不同形状，需要重新编译或使用动态 shape 模式

动态 shape 模式（--input_shape 指定范围）可以支持一定范围内的形状变化，但性能通常不如静态编译。

这是 ATC 方式和 PyTorch 直接运行（动态图模式）的主要区别。

## 4. 什么情况下图优化效果不好

图编译器不是万能的。以下场景可能导致优化效果打折：

### 4.1 自定义算子打断融合

如果模型中有一个 CANN 不认识的自定义操作，图引擎无法识别它的数学特性，就不敢和前后的算子做融合：

```
[MatMul] → [CustomOp] → [Add] → [LayerNorm]

编译器认识 MatMul、Add、LayerNorm，但不认识 CustomOp
→ 只能把 CustomOp 单独执行
→ MatMul 和 Add 之间的融合被打断
```

**解决办法**：用 CANN 支持的标准算子组合替代自定义操作。

### 4.2 动态控制流限制优化

如果模型中有复杂的条件分支（if-else 依赖输入数据），编译器在编译时不知道会走哪个分支，无法做完整的优化。

```python
if input.max() > 0.5:
    x = fast_path(x)
else:
    x = slow_path(x)
```

**解决办法**：尽量把条件判断移到图外面，或者使用 CANN 提供的条件执行原语。

### 4.3 过于碎片化的小算子

如果模型写法导致产生了大量极小的操作：

```python
# 不推荐（产生很多小算子）
x = x + 1
x = x * 2
x = x - 0.5
x = torch.clamp(x, 0, 1)

# 推荐（如果 CANN 有融合版本）
x = optimized_transform(x)  # 一个调用完成
```

小算子太多会导致 Scalar 调度开销大、算子融合困难。

## 5. 图优化和 PyTorch 2.0 torch.compile 的关系

PyTorch 2.0 引入了 `torch.compile`，它的思路和 ATC 类似——捕获计算图，做编译优化。在昇腾上：

- `torch.compile` + torch_npu 后端：动态编译，支持 PyTorch 原生工作流
- ATC 编译为 .om 文件：静态编译，性能通常更好但灵活性低

两者是互补的，不同场景选择不同方式：

| 场景 | 推荐方式 |
|------|---------|
| 研究开发、快速实验 | PyTorch 动态图直接跑 |
| 生产推理、固定模型 | ATC 编译为 .om |
| 需要动态 shape | torch.compile + NPU 后端 |
