# P6C22 — 集群互联：HCCS、RoCE 与通信拓扑

## 引子

当一个 671B 参数的模型（如 DeepSeek-V3）需要数十张卡才能跑时，卡与卡之间的通信速度就成了性能的关键。如果通信太慢，大部分时间花在等待数据交换上，增加再多的卡也没用。

---

## 1. 两级互联架构

昇腾的集群通信分为两级：

### 1.1 机内互联：HCCS

HCCS（Huawei Cache Coherency System）连接同一台服务器内的多颗 NPU（通常 8 颗）。

- **特点**：高带宽、低延迟、缓存一致性
- **对标**：NVIDIA NVLink
- **典型场景**：张量并行（TP）——同一层的计算分到 8 张卡上

### 1.2 机间互联：RoCE

RoCE（RDMA over Converged Ethernet）连接不同服务器之间的 NPU。

- **特点**：基于标准以太网，RDMA 直通显存
- **对标**：NVIDIA InfiniBand
- **典型场景**：数据并行（DP）梯度同步、流水线并行（PP）跨机通信

### 1.3 为什么分两级

机内通信走专用高速通道（HCCS），带宽极高但只能连本机。
机间通信走网络（RoCE），灵活可扩展但带宽低于 HCCS。

设计并行策略时，应该**把通信最频繁的计算（如 TP）放在机内**，把通信较少的计算（如 DP/PP）放在机间。

---

## 2. HCCL：集合通信库

HCCL（Huawei Collective Communication Library）是昇腾的集合通信库，对标 NVIDIA 的 NCCL。

### 2.1 核心通信原语

| 原语 | 操作 | 典型用途 |
|------|------|---------|
| **AllReduce** | 所有卡的数据求和后分发给所有卡 | 数据并行的梯度同步 |
| **AllGather** | 收集所有卡的数据拼接后分发 | 张量并行的结果合并 |
| **ReduceScatter** | 求和后切分分发到各卡 | ZeRO 优化器 |
| **AllToAll** | 所有卡之间多对多交换 | 专家并行（MoE）的路由 |
| **Broadcast** | 一张卡的数据广播给所有卡 | 参数初始化 |

### 2.2 通信效率

通信效率取决于三个因素：
1. **带宽**：每秒能传多少数据
2. **延迟**：第一个字节到达需要多久
3. **通信拓扑**：数据经过几跳才能到达目标

Ring AllReduce 是最常用的通信模式——数据沿环形拓扑传递，每一步每张卡同时发送和接收，带宽利用率接近理论最优。

## 3. 集群拓扑设计

### 3.1 典型拓扑

```
Atlas 800 服务器 (8 × 910B)
├── 机内：8 张 NPU 通过 HCCS 全互联
│   NPU0 ──── NPU1
│   │  \    / │
│   │   \  /  │
│   │    \/   │
│   │    /\   │
│   │   /  \  │
│   NPU2 ──── NPU3
│   (... 类似连接 NPU4-NPU7)
│
└── 机间：通过 RoCE 交换机连接其他服务器
    Server0 ←→ Switch ←→ Server1
                 ↕
              Server2
```

### 3.2 设计原则

- **TP 在机内**：张量并行每一层都要通信，必须走 HCCS
- **PP 尽量机内**：流水线并行只在切分点通信，可以跨机但机内更优
- **DP 可跨机**：数据并行的梯度同步频率较低，可以走 RoCE
- **EP 需要 AllToAll**：MoE 的专家并行通信模式特殊，对带宽要求高

## 本章小结

| 要点 | 内容 |
|------|------|
| 两级互联 | 机内 HCCS（高带宽）+ 机间 RoCE（可扩展） |
| HCCL | 集合通信库，提供 AllReduce 等核心原语 |
| 拓扑设计 | TP 在机内，DP 可跨机，通信密集的计算放近处 |
